## 1. 개요

본 문서는 **Invariant 기반 사고 훈련**을 목표로 하는 Magic Square(4×4) 작업에 대해, **Epic → User Journey → User Stories → Technical 시나리오(Gherkin) → 완성도 점검** 순으로 사용자 여정을 구조화한 기록이다.

선행 문서와의 관계는 다음과 같다.

- **Report 1**: 문제 정의
- **Report 2**: 클린 아키텍처·Dual-Track TDD 설계
- **Report 3**: 개발 환경·Cursor 규칙·ECB·User 엔티티
- **Report 4(본 문서)**: 비즈니스 Epic부터 구현 검증 가능한 시나리오까지의 수직 정렬

## 2. Level 1: Epic — Business Goal (비즈니스 목표)

**Epic**  
Invariant(불변조건) 기반 사고 훈련 시스템 구축

**목적**  
4×4 Magic Square 문제를 학습·실습 매체로 삼아, 다음 역량을 의도적으로 훈련한다.

- **Invariant 중심 설계 사고**: 행·열·대각선 합 등 도메인 불변조건을 설계의 중심에 두고, 구현 전에 “무엇이 항상 참이어야 하는가”를 명시한다.
- **Dual-Track TDD 적용**: 요구·설계 트랙과 구현·리팩터링 트랙을 분리해, 테스트와 코드가 같은 진실의 원천(불변조건·계약)을 향하도록 정렬한다.
- **입력/출력 계약 명확화**: 경계에서의 전제·보장(유효 입력, 실패 시 동작, 결과 의미)을 문서·테스트로 고정해 추측에 의존하지 않는다.
- **설계 → 테스트 → 구현 → 리팩터링 흐름 체화**: 한 사이클을 반복 가능한 규율로 만들어, 변경 시에도 동일한 순서로 품질을 재검증할 수 있게 한다.

**성공 기준**

- **Domain Logic**: 도메인 로직에 대해 테스트 커버리지 **95% 이상**
- **입력 검증**: 입력에 대한 **계약 테스트 100% 통과**
- **하드코딩 없음**: 정답·분기를 우연한 리터럴로 고정하지 않는다.
- **매직 넘버 없음**: 의미 있는 값은 이름(상수·도메인 용어)으로 드러낸다.
- **Invariant → Test 추적 가능**: 핵심 불변조건마다 대응 테스트(또는 그룹)로 거슬러 올라갈 수 있다.

## 3. Level 2: User Journey (사용자 여정) — 예시 1

### 3.1 Persona

- **소프트웨어 개발 학습자**: “동작하는 코드”보다 “설명 가능한 구조”를 목표로 한다.
- **TDD 훈련 중**: Red–Green–Refactor를 습관화하고, 테스트가 요구사항의 스펙 역할을 하도록 익힌다.
- **Clean Architecture 이해 중**: 경계와 의존성 방향을 의식하며 설계한다.

### 3.2 Journey

**Step 1 — 문제 인식**  
“마방진을 구현한다”가 아니라, **“어떤 불변조건과 계약 아래에서 어떤 상태 변환이 일어나는가”**를 먼저 말로·다이어그램으로 고정한다.

**Step 2 — 계약 정의**  
- 입력 스키마 정의  
- 출력 스키마 정의  
- 예외 정책 정의  

**Step 3 — Domain 분리**  
- `BlankFinder` — 빈칸 위치 탐색  
- `MissingNumberFinder` — 누락 숫자 추론·후보 축소  
- `MagicSquareValidator` — 마방진 조건 검증  
- `Solver` — 구성요소 조합으로 해 탐색 또는 불가 판정  

**Step 4 — Dual-Track 진행**  
- UI RED / Logic RED 병렬  
- GREEN 최소 구현  
- REFACTOR 통합 정리  

**Step 5 — 회귀 보호**  
- 엣지 케이스 추가  
- 입력 오류 케이스 추가  
- 조합 실패 케이스 추가

## 4. Level 3: User Stories

### Story 1 — 입력 검증

**As a** 학습자  
**I want** 입력이 정확히 4×4인지 검증되길 원한다  
**So that** 잘못된 데이터가 Domain으로 전달되지 않도록 한다  

**Acceptance Criteria**

- 4×4가 아니면 예외  
- 빈칸이 정확히 2개가 아니면 예외  
- 중복 숫자면 예외  
- 허용 범위(예: 1~16 및 빈칸 표기 규칙) 위반이면 예외  

### Story 2 — 빈칸 탐색

**As a** 학습자  
**I want** 빈칸(예: `0`)의 좌표를 정확히 찾고 싶다  
**So that** 두 빈칸에 대한 조합 시도가 가능하다  

**Acceptance Criteria**

- 좌표는 **row-major** 순서로 정렬·반환  
- 빈칸 좌표는 **정확히 2개** 반환  

### Story 3 — 누락 숫자 탐색

**As a** 학습자  
**I want** 격자에 없는 두 숫자를 알고 싶다  
**So that** 빈칸에 넣을 후보를 확정하고 Solver가 시도 순서를 정할 수 있다  

**Acceptance Criteria**

- 1~16 중 누락된 숫자가 정확히 2개임이 보장된 입력에서 동작  
- 누락 두 숫자는 **오름차순** 반환  

### Story 4 — 마방진 판정

**As a** 학습자  
**I want** 완성된(또는 채워진) 격자가 마방진인지 판정하고 싶다  
**So that** Solver의 후보 배치가 조건을 만족하는지 검증할 수 있다  

**Acceptance Criteria**

- 모든 행의 합 동일  
- 모든 열의 합 동일  
- 두 대각선의 합 동일  

### Story 5 — 두 조합 시도

**As a** 학습자  
**I want** 두 빈칸에 대해 체계적으로 두 번의 배치를 시도하고 싶다  
**So that** 최소 탐색으로 정답을 찾거나, 실패 시 다음 순서로 되돌릴 수 있다  

**Acceptance Criteria**

- 작은 누락 숫자 → 첫 번째 빈칸 순으로 먼저 시도  
- 첫 시도가 조건을 만족하지 않으면 **역순(또는 정책에 따른 reverse 배치)**으로 재시도  
- 정답 표현(또는 시도 결과)에 해당하는 **배열 길이는 6** (프로젝트에서 정의한 6요소 스키마와 일치)

## 5. Level 4: 구현 시나리오 — Technical (Gherkin)

```gherkin
Feature: 4x4 마방진 완성
  불변조건 기반 로직을 검증하기 위해
  TDD를 연습하는 개발자로서
  부분적으로 채워진 4x4 마방진을 완성하고 싶다

  Background:
    Given 4x4 행렬이 주어지고
    And 0은 빈칸을 의미하며
    And 정확히 2개의 셀이 0을 포함하고 있고
    And 숫자는 1부터 16 사이여야 하며
    And 0을 제외한 중복 숫자는 허용되지 않으며
    And 4x4의 마방진 상수는 34이다

  Scenario: 작은 수 → 큰 수 순서로 마방진이 완성되는 경우
    Given 다음과 같은 행렬이 주어졌을 때:
      | 16 |  2 |  3 | 13 |
      |  5 | 11 | 10 |  8 |
      |  9 |  7 |  0 | 12 |
      |  4 | 14 | 15 |  0 |
    When 시스템이 빈칸 좌표를 찾고
    And 누락된 두 숫자를 찾은 뒤
    And 작은 숫자를 첫 번째 빈칸에 배치하고
    And 큰 숫자를 두 번째 빈칸에 배치하면
    Then 모든 행의 합은 34여야 하고
    And 모든 열의 합은 34여야 하며
    And 두 대각선의 합도 34여야 하고
    And 결과는 길이 6의 배열로 반환되어야 하며
    And 반환되는 좌표는 1-index 기준이어야 한다

  Scenario: 역순 배치 시 마방진이 완성되는 경우
    Given 다음과 같은 행렬이 주어졌을 때:
      | 16 |  2 |  3 | 13 |
      |  5 | 11 | 10 |  8 |
      |  9 |  7 |  0 | 12 |
      |  4 | 14 | 15 |  0 |
    When 작은 숫자를 첫 번째 빈칸에 배치했을 때 마방진이 되지 않고
    And 큰 숫자를 첫 번째 빈칸에 배치했을 때 마방진이 되면
    Then 시스템은 역순 배치 결과를 반환해야 하며
    And 최종 행렬은 마방진 상수 34를 만족해야 한다

  Scenario: 빈칸 개수가 올바르지 않은 경우
    Given 행렬에 빈칸이 1개만 존재할 때
    When 유효성 검증을 수행하면
    Then 오류가 발생해야 한다

  Scenario: 중복 숫자가 존재하는 경우
    Given 0을 제외한 중복 숫자가 포함된 행렬일 때
    When 유효성 검증을 수행하면
    Then 오류가 발생해야 한다

  Scenario: 값의 범위를 벗어난 경우
    Given 행렬에 16을 초과하는 숫자가 포함되어 있을 때
    When 유효성 검증을 수행하면
    Then 오류가 발생해야 한다
```

**구현 참고**

- `Background`는 “해결” 시나리오와 “검증 실패” 시나리오가 같은 전제를 공유할 때만 유지하는 것이 안전하다. 실패 시나리오는 별도 `Background` 또는 Examples로 분리하는 것을 권장한다.
- “길이 6 배열”과 “1-index 좌표”는 Step 정의에서 **고정 스키마**(예: `[r1, c1, n1, r2, c2, n2]`)로 명시하는 것이 좋다.

## 6. Level 5: 시나리오 검증 및 정리

### 6.1 시나리오 완성도 체크리스트

#### 4레벨 일관성 확인

**Epic → Journey**

- [x] Epic의 성공 지표가 Journey에 반영됨 (Invariant·계약·Dual-Track·회귀)
- [x] Journey의 모든 단계가 Epic 목표 달성에 기여함
- [x] Pain Point가 명확함 (계약 없이 구현·도메인 혼재·회귀 없는 리팩터 등)

**Journey → Story**

- [x] Journey의 각 Stage마다 최소 1개 Story에 대응됨
- [x] Story가 구체적인 기능으로 변환됨
- [x] Acceptance Criteria가 측정 가능함

**Story → Technical**

- [x] 핵심 AC가 Gherkin `Feature`의 Scenario/Background로 표현됨
- [x] Given–When–Then이 전제·행동·기대로 분리됨
- [x] pytest 등으로 테스트 자동화가 가능한 수준의 단계 granularity

#### Edge Case 커버리지

**정상 케이스**

- [x] Happy Path 시나리오 존재 (작은 수→큰 수 배치로 완성)

**예외 케이스** (CLI 단일 프로세스·도메인 기준)

- [x] 빈칸 개수 위반
- [x] 0 제외 중복 숫자
- [x] 허용 범위 위반 (예: 16 초과)
- [x] 첫 배치 실패 후 역순 시도·최종 판정 (정책이 AC에 추가되면 항목 보강)

**경계 케이스**

- [x] 최솟값·최댓값 (1~16, 0은 빈칸)
- [x] 빈칸 표현(0 두 칸)과 그 외 셀 채움 규칙
- [x] 좌표·정렬 규칙 (row-major, 1-index)

#### 사용자 중심성

**실제 사용자 검증**

- [ ] 학습 페어 또는 리뷰어 1명과 시나리오 리뷰
- [ ] 동일 Persona 기준 Journey 재걸음(Walkthrough)
- [ ] 리뷰 피드백 반영

**감정 흐름**

- [x] 각 Journey Stage마다 기대 감정·전환을 서술할 수 있음
- [x] 불안(계약 불명확) → 안정(계약·테스트 고정) 전환이 Journey 상 연결됨

#### 구현 가능성

**기술 검증**

- [x] pytest·커버리지로 자동 검증 경로 확보 가능
- [x] ECB 디렉터리·의존성 방향과 충돌 없이 모듈 배치 가능
- [x] 마방진 상수(34)·빈칸 2개·길이 6 결과 스키마가 시나리오에 고정됨

**데이터 요구사항**

- [x] 도메인 역할 단위가 Story와 대응됨
- [x] 입출력·예외에 대한 계약 초안이 Story 1·Level 4 Background에 반영됨

## 7. 향후 작업(선택)

- Level 4의 실패 시나리오와 Happy Path의 `Background` 분리 및 Examples 보강  
- “길이 6 배열” 필드 의미를 한 줄로 고정한 **계약 표**(한 페이지) 추가  
- 체크리스트의 미완료 항목(리뷰·Walkthrough) 수행 후 본 문서 갱신
