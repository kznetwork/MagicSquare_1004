# Magic Square (4×4) — Logic / UI Boundary / Data 설계 보고서

| 항목 | 내용 |
|------|------|
| 문서 목적 | Dual-Track UI + Logic TDD 및 Clean Architecture 관점의 **설계·계약·테스트·통합 계획** 기록 (구현 코드 없음) |
| 대상 범위 | Domain Layer, Screen(UI Boundary) Layer, Data Layer, Integration & Verification |
| 작성일 | 2026-04-20 |

**고정 계약 요약**

- **입력:** 4×4 `int[][]`, 0은 빈칸, 빈칸 정확히 2개, 값은 0 또는 1~16, 0 제외 중복 없음.
- **출력:** `int[6]` — `[r1,c1,n1,r2,c2,n2]`, 좌표 **1-index**, `n1,n2`는 누락 두 수이며 (작은 수→첫 빈칸, 큰 수→둘째 빈칸)이 마방진이면 그 순서, 아니면 반대.

# 1) Logic Layer (Domain Layer) 설계

## 1.1 도메인 개념

| 구분 | 이름 | 책임(SRP) |
|------|------|-----------|
| Value Object | `Grid4x4` | 4×4 정수 격자, 좌표·셀 접근, 0=빈칸 의미 보존 |
| Value Object | `CellPosition` | (row,col) 표현 일관성(내부 0-index vs 외부 1-index 중 하나로 통일) 및 변환만 담당 |
| Value Object | `MissingPair` | 누락된 두 숫자 집합 {a,b} (순서 없음) 보관 |
| Value Object | `PlacementProposal` | 두 빈칸에 두 수를 놓는 후보 배치 표현 |
| Entity | `PartialMagicSquare` (동등 개념) | 격자 + 빈칸 2곳 + 채워진 값에 대한 구조/완성 불변조건 검증 진입점 |
| Domain Service | `MagicSquareValidator` | 행·열·두 대각선 합=Magic Constant 판정 |
| Domain Service | `EmptyCellLocator` | 빈칸(0) 정확히 2개 위치 나열(순서 규칙은 유스케이스에서 고정) |
| Domain Service | `MissingNumbersResolver` | 1~16 중 미등장 두 수 도출 |
| Domain Service | `AnswerOrderingPolicy` | 출력 `[r1,c1,n1,r2,c2,n2]`의 (n1,n2) 순서 결정: (작→첫 빈, 큰→둘째) 우선 시도 후 실패 시 반대 |

**의존 방향:** Entity/VO는 서로 및 Validator에만 의존하며, **UI/Data에 의존하지 않음**.

## 1.2 도메인 불변조건(Invariants)

| ID | 불변조건 | 검증 가능 규칙 |
|----|-----------|----------------|
| I1 | 격자 크기는 **4×4** | 행 수=4, 각 행 길이=4 |
| I2 | 값은 **0 또는 1~16** | 각 셀 ∈ {0} ∪ [1,16] |
| I3 | **0은 정확히 2개** | count(0)=2 |
| I4 | **1~16은 각 최대 1회** (0 제외 중복 없음) | ∀v∈[1,16], count(v)≤1 |
| I5 | **완성 마방진**일 때 각 행·열·주대각·부대각 합이 모두 동일 | 4행+4열+2대각=10개 합이 모두 동일 목표값 |
| I6 | **Magic constant** (1~16 표준 4×4)는 **34** | I5의 목표 합으로 고정 |

**분리 원칙:** 부분 입력 단계에서는 I5가 깨질 수 있으므로, **구조 불변(I1~I4)**과 **완성 판정(I5)** 연산을 **별 타입/별 메서드**로 분리해 테스트한다.

## 1.3 핵심 유스케이스(도메인 관점)

| UC | 입력 | 출력 | 실패 |
|----|------|------|------|
| UC-A | `Grid4x4` | 빈칸 두 좌표 `(r1,c1),(r2,c2)` (1-index, 고정 순서 규칙) | 0 개수≠2 |
| UC-B | `Grid4x4` | `MissingPair {a,b}` | 누락이 정확히 두 개가 아님 |
| UC-C | 0 없는 `Grid4x4` | 완성 마방진 여부(boolean 등) | (호출 전제 위반 시 false 또는 명시적 실패 중 하나로 문서 고정) |
| UC-D | 두 빈칸 위치 + 두 수 | 성공 배치에 따른 `int[6]` | 두 배치 모두 I5 불만족 → 해 없음 |

**빈칸 순서 규칙(검증 가능, 고정):** 행 우선 스캔(row 1→4, col 1→4)에서 **먼저 나오는 0**이 첫 빈칸, **나중 0**이 둘째 빈칸. 출력에서 `(r1,c1)`은 첫 빈칸, `(r2,c2)`는 둘째 빈칸.

**숫자 순서 규칙(출력 계약과 정합):** `n_small=min(a,b)`, `n_large=max(a,b)`. 시도1: 첫 빈칸←`n_small`, 둘째←`n_large`가 I5 만족하면 `(n1,n2)=(n_small,n_large)`. 아니면 시도2: 반대 배치가 I5 만족하면 `(n1,n2)=(n_large,n_small)`. 둘 다 실패 시 도메인 실패.

## 1.4 Domain API(내부 계약)

의사 시그니처(구현 코드 아님).

| API | 입력 | 출력 | 실패조건 |
|-----|------|------|----------|
| `parseAndValidateStructure(grid:int[][])` | 외부와 동일 4×4 | `Grid4x4` | I1~I4 위반 |
| `locateTwoZeros(grid:Grid4x4)` | 유효 격자 | `((r1,c1),(r2,c2))` 1-index, 스캔 순서 | 0이 2개가 아님 |
| `findMissingTwoNumbers(grid:Grid4x4)` | 유효 격자 | `MissingPair` | 1~16 등장 수가 “누락 2개”와 불일치 |
| `isCompletedMagicSquare(grid:Grid4x4)` | 격자 | boolean | 전제: 0 없음(위반 시 false 등 문서 고정) |
| `tryFillAndDecideOrder(grid, holes, pair)` | 격자·두 빈칸·누락 쌍 | `int[6]` | 두 배치 모두 실패 |

**DomainError 분류(고정 코드)**

| 코드 | 의미 |
|------|------|
| `SIZE` | 4×4 아님 |
| `RANGE` | 0/1~16 밖 값 |
| `ZERO_COUNT` | 0 개수≠2 |
| `DUPLICATE` | 1~16 중복 |
| `NO_MAGIC_SOLUTION` | 두 배치 모두 실패 |

## 1.5 Domain 단위 테스트 설계(RED 우선)

| TC-ID | 유형 | 입력 요약 | 기대 | 보호 invariant |
|-------|------|-----------|------|----------------|
| D-01 | 정상 | 완성 마방진에서 2칸 0으로 바꾸고 누락 수 일치 | `int[6]`로 복원 시 I5 만족 | I5 |
| D-02 | 정상 | 한쪽 배치만 성공하도록 설계 | (n1,n2)가 규칙대로 | 출력 순서 규칙 |
| D-03 | 정상 | (선택) 두 배치 성공 가능 입력이 도메인 정책상 금지면 생략, 존재하면 유일 규칙 테스트 | 단일 기대값 | UC-D |
| D-04~D-05 | 비정상 | 0이 1개 / 3개 | `ZERO_COUNT` | I3 |
| D-06~D-07 | 비정상 | 17 / 음수 | `RANGE` | I2 |
| D-08 | 비정상 | 1~16 중복 | `DUPLICATE` | I4 |
| D-09~D-10 | 비정상 | 크기 불일치·가변 행 | `SIZE` | I1 |
| D-11 | 엣지 | 두 배치 모두 합 34 실패 | `NO_MAGIC_SOLUTION` | I5 |
| D-12 | 엣지 | 빈칸 스캔 순서 고정 검증 | (r1,c1)이 먼저 | 빈칸 순서 규칙 |
| D-13 | 단위 | 0 남은 격자로 완성 판정 | `false` 고정 | I5 전제 |
| D-14 | 단위 | 완성 마방진 | `true` | I5 |

**RED 권장 순서:** (1) I5 판정 D-14,D-13 → (2) 누락 수 D-01 하위 → (3) 빈칸 D-12 → (4) 최종 조합 D-01~D-03 → (5) 구조 오류 D-04~D-10.

# 2) Screen Layer (UI Layer) 설계 (Boundary Layer)

## 2.1 사용자/호출자 관점 시나리오

| 단계 | 행위 | 결과 |
|------|------|------|
| S1 | 호출자가 4×4 int[][] 제공 | Boundary가 raw 입력 수용 |
| S2 | Boundary가 스키마/구조 검증 | 통과 시 Domain 호출 준비 |
| S3 | Domain에 위임 | 성공 시 int[6], 실패 시 도메인 오류 코드 |
| S4 | Boundary가 출력 스키마로 변환 | 성공/표준 에러 객체 |

## 2.2 UI 계약(외부 계약)

### Input schema

| 필드 | 타입 | 규칙 |
|------|------|------|
| `grid` | int[][] | 길이=4, 각 행 길이=4 |
| 셀 값 | int | 0 또는 1~16 |
| zeros | 파생 | 정확히 2 |
| 1~16 | 파생 | 중복 없음 |

### Output schema (성공)

| 필드 | 타입 | 규칙 |
|------|------|------|
| `answer` | int[6] | `[r1,c1,n1,r2,c2,n2]` |
| 좌표 | int | 1-index, 1~4 |
| n1,n2 | int | 누락 두 수, 순서는 문제 계약대로 |

### Error schema

| code | 의미 |
|------|------|
| `INVALID_SIZE` | 4×4 아님 |
| `INVALID_VALUE_RANGE` | 값 범위 위반 |
| `INVALID_ZERO_COUNT` | 0 개수≠2 |
| `DUPLICATE_VALUES` | 1~16 중복 |
| `NO_SOLUTION` | 도메인 해 없음 |

**검증 책임 중복 방지(택일, 문서에 하나만 선택):**

- **정책 A(추천):** Boundary는 null/길이 등 최소 검증, 구조·중복·0개수·범위는 **Domain 단일 진실원천**.
- **정책 B:** Boundary에서 전부 검증, Domain은 불변 위반 불가 입력만 받음.

## 2.3 UI 레벨 테스트(Contract-first, RED 우선)

Domain은 Mock/Stub 가정.

| UI-TC | 내용 | 기대 | Mock |
|--------|------|------|------|
| U-01 | 3×4 등 잘못된 크기 | `INVALID_SIZE` | Domain 0회 |
| U-02 | null grid / null row | `INVALID_SIZE` (코드 통일) | 0회 |
| U-03 | 빈칸 1개 | `INVALID_ZERO_COUNT` | 정책 A면 Domain 또는 Boundary 일관 |
| U-04 | 빈칸 3개 | `INVALID_ZERO_COUNT` | 동일 |
| U-05 | 값 99 등 | `INVALID_VALUE_RANGE` | 동일 |
| U-06 | 중복 | `DUPLICATE_VALUES` | 동일 |
| U-07 | 정상 입력 | int[6] | Mock 고정 반환 |
| U-08 | 반환 포맷 | 길이 6, 좌표 1~4, n1≠n2, 범위 1~16 | U-07 확장 |
| U-09 | Domain NO_SOLUTION | `NO_SOLUTION` | Mock 실패 |

**Contract-first 체크리스트**

- [ ] 성공 응답에 `answer:int[6]` (또는 언어 관례 동등 키) 존재.
- [ ] 실패 응답에 `code` + `message` 존재(문구는 2.4).
- [ ] 조기 실패 시 Domain 호출 횟수 **정확히 0** 검증(U-01 등).

## 2.4 UX/출력 규칙

**에러 메시지 표준(정확 문자열)**

| code | message |
|------|---------|
| `INVALID_SIZE` | `Grid must be 4x4.` |
| `INVALID_VALUE_RANGE` | `Each cell must be 0 or 1..16.` |
| `INVALID_ZERO_COUNT` | `There must be exactly two zeros (empty cells).` |
| `DUPLICATE_VALUES` | `Values 1..16 must not duplicate (excluding zeros).` |
| `NO_SOLUTION` | `No placement makes a 4x4 magic square with magic sum 34.` |

**성공 출력(텍스트 변환이 있을 경우):** 한 줄 `answer: [r1, c1, n1, r2, c2, n2]` — 구분자·공백 규칙을 구현 시 동일 적용.

# 3) Data Layer 설계

## 3.1 목적 정의

| 항목 | 내용 |
|------|------|
| 필요성 | 학습용: 저장소 구현 교체(메모리/파일)로 **재현성·테스트 격리** 확보 |
| 범위 | 입력 행렬 + (선택) 마지막 성공 결과 수준, DB/마이그레이션 없음 |

## 3.2 인터페이스 계약

| 메서드(의사) | 동작 | 저장 대상 | 실패 |
|---------------|------|-----------|------|
| `MatrixRepository.saveSession(inputGrid, outputAnswerOptional)` | 세션 저장 | 4×4 입력, 선택적 int[6] | 쓰기/직렬화 실패 |
| `MatrixRepository.loadLastSession()` | 마지막 세션 로드 | — | 없음/손상 |
| `MatrixRepository.clear()` (선택) | 초기화 | — | — |

**SessionRecord 필드:** `grid:int[][]`, `answer:int[6]?`, `savedAt`(시각 타입은 구현에서 고정).

## 3.3 구현 옵션 비교

| 옵션 | 장점 | 단점 |
|------|------|------|
| A InMemory | 빠름, 파일 의존 없음 | 프로세스 종료 시 소실 |
| B File(JSON) | 사람이 읽기 쉬움 | IO·권한 flake |
| B' File(CSV) | 단순 표현 | 2차원·이스케이프 규칙 검증 비용 큼 |

**추천:** **InMemory 기본 + JSON 파일 어댑터** 병행. 통합 회귀는 InMemory로 안정화, 파일은 IO 실패·손상 시나리오를 별도 테스트로 격리.

## 3.4 Data 레이어 테스트

# 4) Integration & Verification

## 4.1 통합 경로 정의

**의존성 방향:** `Boundary` → `Application(선택)` → `Domain` ; `Application` → `Data` 인터페이스 ; **Domain은 UI/Data 비의존.**

| 경로 | 설명 |
|------|------|
| R1 | Boundary 검증 → Domain.solve → Boundary 출력 |
| R2 | Application: solve 후 Repository.save |
| R3 | loadLast → Boundary 출력(조회 기능이 있을 때만) |

## 4.2 통합 테스트 시나리오

**정상(2+)**

| ID | 내용 | 검증 |
|----|------|------|
| IT-OK-01 | 유효 입력 E2E + InMemory repo | int[6] + 저장 세션 일치 |
| IT-OK-02 | 서로 다른 입력 2회 연속 | 마지막 세션이 최종본(정책 고정) |

**실패(3+)**

| ID | 유형 | 검증 |
|----|------|------|
| IT-FAIL-01 | 입력 오류(0 개수) | Domain 미호출 + code/message 고정 |
| IT-FAIL-02 | Domain 실패(Mock) | `NO_SOLUTION` + 저장 정책 문서대로 |
| IT-FAIL-03 | 저장 IOException 시뮬 | `PERSISTENCE_ERROR` 등 래핑 고정 |

## 4.3 회귀 보호 규칙

| 규칙 | 내용 |
|------|------|
| 계약 | `int[6]`, 1-index, (n1,n2) 규칙 변경 시 버전 없이는 금지 |
