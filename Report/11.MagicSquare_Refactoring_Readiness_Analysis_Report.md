## 1. 테스트 현황 및 리팩토링 전제

### 1.1 `test_*.py` 존재 여부

다음 등 **`test_*.py`는 존재한다.**

| 경로 | 비고 |
|------|------|
| `tests/magicsquare/test_magic_square_logic_red.py` | `magicsquare.domain` (`find_blank_coords` 등) import |
| `tests/magicsquare/test_magic_square_ui_boundary_red.py` | `magicsquare.boundary` (`validate` 등) import |
| `tests/entity/models/test_user.py` | 엔티티 샘플 테스트 |

다만 **마방진 도메인·경계** 관점에서는:

- **Logic RED**·**UI/Boundary RED** 파일에 `pytest.fail("RED")` 스켈레톤이 많아, **실제 assertion으로 동작을 고정한 테스트는 부족**하다.
- **`gui/main_window.py`**에 대해서는 `tests` 아래에서 **직접적인 import·테스트가 사실상 없다.**

### 1.2 테스트 없이 리팩토링을 시작하면 안 되는 이유 (한 줄)

**리팩토링으로 바뀐 동작이 맞는지 자동으로 잡아줄 안전망이 없어서, 회귀를 사람이 수동으로만 믿게 되고 실수 비용이 커지기 때문이다.**

## 3. ECB(Entity–Control–Boundary) 분석 요약

| 파일 | ECB 역할 요약 |
|------|----------------|
| `domain.py` | **상태를 들인 Entity 클래스는 없고**, `list[list[int]]`에 대한 **순수 도메인 연산(규칙 실행)**이 중심이다. 사용자 구분(데이터 보관 vs 규칙 실행)으로는 **Control(비즈니스 규칙·연산 실행)**에 가깝다. |
| `boundary.py` | 외부 입력에 `ValueError`·고정 메시지로 검증하고 도메인을 호출한다. **UI와 도메인 사이의 응용 경계(검증·오케스트레이션·예외 매핑)**로 Boundary 역할에 부합한다. |
| `gui/main_window.py` | 위젯·이벤트·표현이 주이며 도메인 규칙을 풀어 쓰지 않는다. **이중 `validate` 호출**은 레이어 혼입이라기보다 **중복 호출**에 가깝다. |

**잘못된 레이어 배치:** 명백한 위반은 없다(도메인에 Qt 없음, GUI에 마방진 판정 없음).

**`domain.py` 내부 Entity vs Control이 논의될 만한 경계:** `solve_placement`는 빈 칸·빠진 수 조회 후 두 배치를 시도하는 **유스케이스 조율(Control 성격)**에 가깝고, `find_blank_coords`, `find_missing_pair_sorted`, `is_completed_magic_square`, `_copy_grid`는 **그리드에 대한 규칙·질의**에 가깝다. 클래스 단위 ECB를 엄격히 적용할 경우 **`solve_placement`를 먼저 분리하기 쉬운 경계**로 볼 수 있다.

## 4. SRP(단일 책임 원칙) 위반 후보

### 4.1 하나의 함수가 두 가지 이상의 역할

| 파일명:줄번호 | 요약 |
|---------------|------|
| `domain.py:17-33` | 스캔으로 0 위치 탐색 + 1-based 좌표 반환. |
| `domain.py:54-75` | 빈 칸(0) 없음 확인 + 모든 마법선 합 검사. |
| `domain.py:78-106` | 조회·배치·판정·결과 조립·예외까지 한 흐름. |
| `boundary.py:18-57` | I1~I4 독립 규칙을 한 함수가 전부 수행. |
| `boundary.py:60-73` | 검증 위임 + 풀이 + 예외 매핑. |

### 4.2 클래스가 데이터 보관과 검증을 동시에 담당

**해당 없음** (`NoMagicSolutionError`는 예외 타입만 존재).

### 4.3 `gui/main_window.py`에서 UI 외 비즈니스 판단(예: `total == 34`)

**해당 없음.** `validate`/`solve` 위임만 수행한다.

### 5.2 테스트 선행 필요 항목

리팩토링 전에 **실제 assertion(GREEN)**으로 고정할 것을 권장하는 함수 목록:

- `boundary.validate` (분기별 `ValueError` 및 `MSG_*` 메시지)
- `boundary.solve` (성공 시 6원소, `NoMagicSolutionError` → `ValueError(MSG_NO_SOLUTION)` 및 `__cause__`)
- `domain.solve_placement`, `domain.is_completed_magic_square`
- `domain.find_blank_coords`, `domain.find_missing_pair_sorted` (RED 스켈레톤을 GREEN으로 보강)
- `gui/main_window.py` (선택: `pytest-qt` 등 또는 수동 스모크)

### 5.3 리팩토링 후 검증 방법

**회귀 테스트 실행 (프로젝트 루트):**

```bash
pytest
```

마방진 모듈만:

```bash
pytest tests/magicsquare
```

**외부 동작 불변 확인:**

1. 위 테스트가 전부 통과하고, 특히 메시지·예외 타입이 기존과 동일한지 확인한다.
2. GUI 수동 확인: 대표 입력(유효·무효·해 없음)에 대해 경고 문구·결과 라벨 형식(쉼표 구분 6개 숫자)이 이전과 동일한지 확인한다.
