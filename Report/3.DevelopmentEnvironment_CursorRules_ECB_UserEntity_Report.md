## 1. 작업 개요

본 문서는 **문제 정의(Report 1)** 및 **클린 아키텍처·듀얼 트랙 TDD 설계(Report 2)** 이후에 진행된 **저장소 개발 환경·AI 규칙·첫 도메인 엔티티** 관련 변경을 한곳에 정리한다.

핵심 성과는 다음과 같다.

- 프로젝트 루트 **`.cursorrules`**에 MagicSquare 전용 정책(YAML)을 정의·보강했다.
- Cursor 권장 형식인 **`.cursor/rules/*.mdc`**로 규칙을 주제별로 분할해 에이전트가 읽기 쉬운 구조로 옮겼다.
- ECB 권장 디렉터리에 맞춰 **`src/magicsquare/entity`**에 **`User`** 엔티티를 추가하고, **`tests/entity/models`**에 pytest 기반 테스트를 배치했다.
- **`pyproject.toml`**으로 `src` 레이아웃과 pytest `pythonpath`를 설정했다.

## 2. Cursor 규칙 (`.cursorrules`)

### 2.1 구성

루트 `.cursorrules`는 YAML 형식이며, 최상위 섹션은 다음을 포함한다.

| 섹션 | 요약 |
|------|------|
| `project` | MagicSquare 목표·언어(Python)·ECB·TDD 방향 |
| `code_style` | Python 3.10+, PEP8, Black 88, 타입 힌트·Google docstring 등 |
| `architecture` | ECB 3레이어 정의 및 의존성 방향(안쪽 도메인 향함) |
| `tdd_rules` | Red / Green / Refactor 단계별 `description`, `rules`, `must_not` |
| `testing` | pytest, AAA, 커버리지 최소 80%, fixture 스코프 규칙, `test_` 접두사 |
| `forbidden` | `print()` 남용, 하드코딩, `except` 남용 등 패턴·이유·대안 |
| `file_structure` | ECB 기준 권장 트리(boundary / control / entity / tests) |
| `ai_behavior` | 코드 작성 전·중·후 행동 및 위반 시 경고 출력 |

섹션 사이에는 80자 `#` 구분선 주석이 있다. (YAML 주석으로 파서에는 남지 않음.)

### 2.2 `.cursor/rules/*.mdc` 분할

Cursor 권장 규칙 위치인 **`.cursor/rules/`**에 아래 **5개** `.mdc` 파일을 두었다.

| 파일 | 적용 방식 | 내용 |
|------|-----------|------|
| `magicsquare-project.mdc` | `alwaysApply: true` | 프로젝트 목표, AI 전·중·후 행동, 위반 시 경고 |
| `magicsquare-python-code-style.mdc` | `globs: "**/*.py"` | PEP8, Black, 타입힌트, docstring, import |
| `magicsquare-ecb-architecture.mdc` | `alwaysApply: true` | Boundary / Control / Entity, 의존성, 디렉터리 트리 |
| `magicsquare-tdd-testing.mdc` | `globs: "**/*.py"` | TDD 3단계, pytest·AAA·커버리지·fixture |
| `magicsquare-forbidden.mdc` | `alwaysApply: true` | 금지 패턴과 대안 |

루트 `.cursorrules` 상단에는 **기준 규칙이 `.mdc`에 있음**을 안내하는 주석을 추가했다. 루트 YAML은 동일 정책의 **미러(참고·백업)** 로 유지할 수 있으며, 중복 로딩이 부담되면 한쪽만 남기는 정리가 가능하다.

## 3. Python 패키지 및 ECB 소스 트리

### 3.1 `pyproject.toml`

- **빌드**: `setuptools`, 패키지 탐색 경로 `src/`
- **런타임**: `requires-python >= 3.10`, 의존성은 현재 비움
- **개발**: optional `dev`에 `pytest>=7.0`
- **pytest**: `testpaths = ["tests"]`, `pythonpath = ["src"]` 로 `magicsquare` 패키지 import 가능

### 3.2 현재 구현된 디렉터리

이번 작업에서 **실제 파일이 생긴** ECB 관련 경로는 다음과 같다.

```text
src/magicsquare/
├── __init__.py
└── entity/
    ├── __init__.py
    └── models/
        ├── __init__.py      # User, UserValidationError 재노출
        └── user.py          # User 엔티티

tests/
└── entity/
    └── models/
        └── test_user.py     # User 단위 테스트
```

`boundary/`, `control/` 폴더는 보고서·규칙 문서상 권장 구조로만 정의되어 있으며, **아직 빈 디렉터리로 생성하지는 않았다.** (필요 시 동일 규칙으로 추가하면 된다.)

## 4. `User` 엔티티 (entity 레이어)

### 4.1 설계 요지

- **레이어**: ECB의 **entity** — 도메인 데이터·불변식만 담당. `control`·`boundary`에 의존하지 않음.
- **구현**: `@dataclass(frozen=True, slots=True)` 로 불변 객체.
- **불변식**: `user_id`·`display_name`은 공백 제거 후 비어 있으면 안 됨. `email`은 공백 제거 후 소문자 정규화 및 정규식 기반 형식 검증.
- **변경 API**: 가변 필드 대신 `with_display_name`, `with_email`으로 **새 인스턴스** 반환.
- **예외**: 도메인 위반 시 `UserValidationError` ( `ValueError` 서브클래스 ).

### 4.2 문서화·스타일

- 모든 public 메서드에 **Google 스타일** docstring (`Args` / `Returns` / `Raises`).
- **타입 힌트**를 파라미터·반환값에 명시.

## 5. 테스트 (pytest)

- **위치**: `tests/entity/models/test_user.py`
- **규칙**: AAA(Arrange–Act–Assert), 함수명 `test_` 접두사.
- **검증 범위**: 정상 생성·필드 정규화, 빈 `user_id` / `display_name` / `email`, 잘못된 이메일 형식, `with_display_name`·`with_email` 불변성 및 필드 유지.

**실행 결과(작업 시점):** `7 passed` (로컬 `pytest` 기준).

## 7. 후속 권장 작업

1. **`control` 레이어**: 사용자 등록·조회 등 유스케이스가 생기면 `entity`만 사용하는 서비스/인터랙터 추가.
2. **`boundary` 레이어**: CLI/API에서 DTO ↔ `User` 매핑 및 검증 실패 메시지 변환.
3. **품질 게이트**: CI에서 `pytest`, (선택) `ruff`/`mypy`, 커버리지 80% 이상 자동 검증.
4. **규칙 파일 정리**: `.cursorrules`와 `.mdc` 중복이 부담되면 한쪽을 단일 소스로 정하고 다른 쪽은 링크·요약만 유지.
