## 부록. Dual-Track TDD와 맞춘 깃 브랜치 요약(참고)

프로젝트 방법론(**Dual-Track UI + Logic TDD** → **GREEN** → **Dual-Track Refactoring**)과의 정렬을 위해 다음을 권장한다.

- **장수 브랜치**: `main`(안정)·`develop`(통합, 선택 운용 시).  
- **작업 브랜치**: `feat/<scope>-<slug>` 등 짧은 수명; 한 브랜치 안에서 Red(듀얼 트랙 테스트) → Green → Refactor를 **커밋 경계**로 나누어 기록하면 리뷰·되돌리기가 쉽다.  
- **머지**: 팀 규칙에 따라 squash 또는 merge commit 선택.

본 부록은 워크플로 가이드이며, GitHub의 실제 브랜치 보호 규칙은 저장소 정책에 따른다.
