# 1. Executive Summary

Magic Square (4×4)는 **알고리즘 경쟁이 아니라** TDD·경계 설계·불변조건 기반 검증 사고를 반복 훈련하기 위한 제품이다. 호출자는 4×4 정수 행렬을 제공하고, 시스템은 **고정된 입력/출력 계약**에 따라 두 빈칸(0)을 채워 완성 4×4 마방진(행·열·주·부 대각선 10개 선의 합이 34)을 만족하는 배치를 찾거나, 계약상 불가능·구조 위반을 **코드와 메시지가 고정된 실패**로 반환한다. **그래픽 UI·웹·DB 없이** Boundary(입출력·검증·오류 매핑)와 Domain(순수 판정·탐색·해 결정)의 Dual-Track TDD로 개발하며, 모든 요구사항은 자동 테스트로 검증 가능한 문장으로만 기술한다.

---

# 2. Problem Statement (문제 정의)

## 2.1 정확한 문제 정의(제품이 닫는 것)

표면적 정의인 “4×4에 숫자를 채워 마방진을 만든다”는 **계약이 비어** 동일한 문장으로 서로 다른 문제를 풀 수 있다. 본 제품이 다루는 문제는 다음이다.

- **고정 크기(4×4)** 격자에, 합의된 **숫자 집합·빈칸·중복 규칙(I1~I4)**을 적용한다.
- “합을 비교해야 하는 집합”은 **4행·4열·주대각·부대각**으로 명시되며, 완성 판정 시 각 집합의 합은 **동일한 상수 34**와 같아야 한다(I5~I6).
- 출력은 **완전 배치를 6원소 벡터로 압축한 표현**이거나, 구조 위반·해 없음은 **계약에 정의된 실패 범주**로만 반환한다.

즉 제품은 “마방진을 그린다”가 아니라, **불변조건과 입출력 계약을 검증 가능한 형태로 만족시키거나, 만족 불가를 분류해 반환한다**.

## 2.2 입력/출력 계약이 핵심인 이유

입력이 흐리면 동일 절차가 다른 문제를 풀게 되고, 출력이 흐리면 “거의 맞음”, “여러 해”, “좌표 기준 불일치” 등 **해석 분쟁**이 생긴다. 명확한 입출력은 문서 장식이 아니라 **판정 기계를 성립시키는 전제**이며, 테스트는 계약·불변·판정을 구현 중에도 고정하는 외부 장치다(Report/1 STEP 4~5, Report/4 Level 1 성공 기준과 정합).

# 4. Scope

## 4.1 In-Scope

- 4×4 입력 행렬에 대한 **구조 검증(I1~I4)** 및 실패 응답
- **행 우선(row-major)** 스캔으로 정의되는 **첫 번째·두 번째 빈칸(0) 좌표** 산출(1-index)
- 1~16에서 **누락된 정확히 두 수** 산출
- 0이 없는 격자에 대한 **완성 마방진 판정(I5~I6)**
- 누락 두 수를 두 빈칸에 놓는 **두 번의 시도(정방향·역배치)** 및 `int[6]` 성공 응답, 또는 **해 없음** 실패 응답
- Dual-Track TDD에 필요한 **계약 테스트·도메인 단위 테스트·회귀 정책**

## 4.2 Out-of-Scope

- **그래픽 UI** 화면, 라우팅, 프론트엔드 자산
- **DB** 저장·조회·마이그레이션, 파일 영속화 세션
- **N×N 일반화**(n≠4): 본 PRD의 기능 요구사항 범위 밖이며, 향후 별도 확장 PRD가 필요하다
- 마방진 **전수 생성**, 다해 탐색·대칭 해 정규화·최적화 경쟁
- 네트워크 API, 인증·권한

# 5. Functional Requirements (기능 요구사항)

**공통 입력 타입(참조):** `grid`는 `int[][]`이다.  
**공통 출력 타입(성공):** `int[6]` — `[r1, c1, n1, r2, c2, n2]` (좌표 1-index).  
**공통 실패 타입:** Boundary가 노출하는 `code` 문자열과 `message` 문자열(§7, §8.1 표와 **문자 단위 동일**).

## FR-01 입력 검증 (Boundary)

| 항목 | 내용 |
|------|------|
| **Feature ID** | FR-01 |
| **설명** | Boundary는 외부가 제공한 `grid`에 대해 I1~I4를 검사하고, 위반 시 표준 실패를 반환한다. |
| **입력** | `int[][] grid` |
| **처리 규칙** | 다음이 모두 참이어야 하며, 하나라도 거짓이면 해당하는 **최우선 규칙 하나**에 매핑된 오류로 실패한다(검사 순서는 BR-10). (1) `grid` 및 각 행이 `null`이 아니다. (2) I1: 행 수 4, 각 행 길이 4. (3) I2: 각 셀 ∈ {0} ∪ [1,16]. (4) I3: 값 0인 셀의 개수는 정확히 2. (5) I4: ∀v∈[1,16], 값 v인 셀의 개수는 최대 1. |
| **출력** | 성공 시 Domain 해 결정 경로로의 위임에 필요한 내부 표현(구현체 선택) 또는 실패 시 `code`·`message`. |
| **승인 기준(AC)** | AC-FR-01-01: `grid`가 `null`이면 Boundary는 `code=INVALID_SIZE`이고 `message`가 §8.1 표의 `INVALID_SIZE` 문구와 동일한 실패를 반환하며, **Domain 해 결정 진입점(`resolve` 동등)**은 **0회** 호출된다(테스트 스파이·모크로 호출 횟수 검증). AC-FR-01-02: 외부 길이가 4가 아니거나 임의 행이 길이 4가 아니면 `INVALID_SIZE` 동일 조건. AC-FR-01-03: 0의 개수가 2가 아니면 `INVALID_ZERO_COUNT`. AC-FR-01-04: 1~16 밖 값이 존재하면 `INVALID_VALUE_RANGE`. AC-FR-01-05: 1~16 중복이 존재하면 `DUPLICATE_VALUES`. |
| **관련 오류/예외 정책** | §8.1 코드·메시지 표. 구조 실패 시 **해 결정 진입점은 호출 금지**. |

## FR-02 빈칸 탐색

| 항목 | 내용 |
|------|------|
| **Feature ID** | FR-02 |
| **설명** | I3가 참인 유효 격자에서 값 0인 두 셀의 위치를 **행 우선 스캔 순서**로 식별한다. |
| **입력** | I1~I4를 만족하는 `grid`의 내부 표현 |
| **처리 규칙** | 행 인덱스 1→4, 열 인덱스 1→4 순서로 스캔할 때 **처음 만나는 0**이 `(r1,c1)`, **두 번째 0**이 `(r2,c2)`이다(BR-04). |
| **출력** | 정수 좌표 `(r1,c1,r2,c2)` (각 1~4) |
| **승인 기준(AC)** | AC-FR-02-01: TQ-01에 대해 `(r1,c1)=(3,4)`, `(r2,c2)=(4,4)`이다. AC-FR-02-02: 두 좌표는 서로 다르다. AC-FR-02-03: 스캔 순서를 바꾼 정의(열 우선 등)로 구현하지 않는다(고정 문서 위반으로 간주). |
| **관련 오류/예외 정책** | I3 불만족 입력은 FR-01에서 종료되므로 본 기능의 전제는 “0이 정확히 2개”이다. |

## FR-03 누락 숫자 탐색

| 항목 | 내용 |
|------|------|
| **Feature ID** | FR-03 |
| **설명** | 1~16 중 격자에 나타나지 않은 **정확히 두 수** `{a,b}`를 산출한다. |
| **입력** | I1~I4를 만족하는 `grid` |
| **처리 규칙** | multiset 관점에서 1~16 각각 정확히 한 번씩 등장해야 하며, 0 두 칸이 비움을 표시한다. 누락 집합은 `{a,b}`로 유일하게 결정된다(BR-05). |
| **출력** | 내부 표현 `MissingPair` 또는 동등물: 원소 두 개, 순서 무관 저장 |
| **승인 기준(AC)** | AC-FR-03-01: TQ-01에 대해 누락 집합은 `{1,6}`이다. AC-FR-03-02: TQ-NOSOL에 대해 누락 집합은 `{15,16}`이다. |
| **관련 오류/예외 정책** | I1~I4 전제 하에서 누락이 2개가 아닌 경우는 **논리적으로 발생하지 않는다**. 발생 시 구현 버그로 분류하며, 본 PRD 범위 밖의 복구 요구사항은 정의하지 않는다. |

## FR-04 마방진 판정

| 항목 | 내용 |
|------|------|
| **Feature ID** | FR-04 |
| **설명** | 0이 없는 완전 채움 격자에 대해 I5~I6를 검사한다. |
| **입력** | 0이 없는 4×4 정수 격자(내부 표현) |
| **처리 규칙** | 4행·4열·주대각·부대각 **총 10개 선** 각각의 합을 계산한다. 10개 합이 모두 34이면 참, 하나라도 다르면 거짓(BR-06, BR-07). |
| **출력** | boolean(또는 동등한 이진 판정) |
| **승인 기준(AC)** | AC-FR-04-01: TQ-01에 두 빈칸을 FR-05 **성공 반환과 동일한** `(n1,n2)`로 채운 완전 격자는 `true`이다. AC-FR-04-02: TQ-01에 두 빈칸을 FR-05 **성공 반환과 다른** `(n1,n2)`로 채운 완전 격자는 `false`이다(동일 좌표 `(r1,c1),(r2,c2)` 유지). AC-FR-04-03: TQ-NOSOL에서 0 두 칸을 `{15,16}`로 채운 완전 격자는 `false`이다. |
| **관련 오류/예외 정책** | 0이 남아 있는 격자에 대한 판정 결과는 **호출 전제**로 문서화한다(BR-08): 본 제품의 운영 경로에서는 FR-05가 임시 격자를 만들어 0 없이 판정한다. |

## FR-05 해 찾기(solution): 두 조합 시도 및 반환

| 항목 | 내용 |
|------|------|
| **Feature ID** | FR-05 |
| **설명** | FR-02·FR-03 결과를 사용해 두 빈칸에 누락 두 수를 놓는 **최대 두 번의 시도** 후 `int[6]` 또는 해 없음 실패를 반환한다. |
| **입력** | I1~I4 만족 `grid`, FR-02 좌표, FR-03 누락 `{a,b}` |
| **처리 규칙** | `n_small=min(a,b)`, `n_large=max(a,b)` (BR-09). **시도1:** 첫 빈칸에 `n_small`, 둘째 빈칸에 `n_large`를 배치한 완전 격자에 대해 FR-04가 참이면 성공으로 `[r1,c1,n_small,r2,c2,n_large]` 반환. **시도2:** 시도1이 실패한 경우에만, 첫 빈칸에 `n_large`, 둘째에 `n_small`을 배치해 FR-04가 참이면 `[r1,c1,n_large,r2,c2,n_small]` 반환. **시도1·시도2 모두 실패**하면 Boundary는 `NO_SOLUTION`을 반환한다(BR-03, §8.1 메시지). |
| **출력** | 성공: `int[6]`; 실패: `NO_SOLUTION` |
| **승인 기준(AC)** | AC-FR-05-01: TQ-01에 대해 반환 배열은 길이 6이고, 모든 좌표는 1~4이며, `{n1,n2}={1,6}`이다. AC-FR-05-02: TQ-01에 대해 FR-04 기준 완성 마방진이 되는 `(n1,n2)` 순서 **하나만** 존재하므로, 반환 배열은 그 순서와 **원소 단위로 일치**한다. AC-FR-05-03: TQ-NOSOL에 대해 Boundary는 `code=NO_SOLUTION`이고 `message`가 §8.1 표와 동일하다. AC-FR-05-04: 성공·실패 모든 경로에서 **입력 `grid`의 모든 셀 값은 호출 전후 동일**하다(§7 부작용 금지). |
| **관련 오류/예외 정책** | `NO_SOLUTION`은 구조 오류가 아니다. Domain 내부 코드 `NO_MAGIC_SOLUTION`이 있을 경우 Boundary는 이를 `NO_SOLUTION`으로 매핑한다(Report/2 표와 정합). |

# 6. Business Rules (도메인 규칙)

| ID | 규칙(항상 참이어야 하는 진술) |
|----|--------------------------------|
| BR-01 | 격자의 행 수는 4이고, 각 행의 길이는 4이다(I1). |
| BR-02 | 각 셀의 값은 0 또는 1 이상 16 이하의 정수이다(I2). |
| BR-03 | 값 0인 셀은 정확히 2개이다(I3). |
| BR-04 | 첫 번째 빈칸은 행 우선 스캔에서 먼저 발견되는 0의 좌표이고, 두 번째 빈칸은 나중에 발견되는 0의 좌표이다. |
| BR-05 | 1~16의 각 값은 0을 제외한 셀에서 최대 1회 등장한다(I4). |
| BR-06 | 완성 마방진에서 4행·4열·주대각·부대각 **10개 선**의 합은 서로 같다(I5). |
| BR-07 | 본 제품에서 그 공통 합은 34이다(I6). |
| BR-08 | FR-04는 **0이 존재하지 않는** 완전 격자에만 적용한다. |
| BR-09 | FR-05의 `n_small`, `n_large`는 누락 두 수에 대해 `n_small≤n_large`를 만족한다. |
| BR-10 | FR-01 검증 순서는 다음이다: `null`·행 길이(I1) → 값 범위(I2) → 0 개수(I3) → 1~16 중복(I4). 동시 위반 시 **표에 더 위에 있는 코드**를 반환한다: `INVALID_SIZE` → `INVALID_VALUE_RANGE` → `INVALID_ZERO_COUNT` → `DUPLICATE_VALUES`. |

# 7. Non-Functional Requirements

| ID | 요구사항 | 검증 방법 |
|----|-----------|-----------|
| NFR-01 | Domain Logic 라인 커버리지 **≥ 95%** | `pytest-cov` 또는 동등 도구로 수치 리포트에 `≥ 0.95` 표기 |
| NFR-02 | Boundary(입력 검증·출력·오류 매핑) 라인 커버리지 **≥ 85%** | 동일 |
| NFR-03 | 동일 입력에 대해 성공 경로 출력 배열은 **항상 동일**하고, 동일 구조 위반 입력에 대해 실패 `code`는 항상 동일하다 | 결정론 테스트: 픽스처 입력 3회 반복 실행 |
| NFR-04 | 공개 API는 호출자가 소유한 입력 `int[][]`의 어떤 셀도 변경하지 않는다 | 호출 전 깊은 복사본과 호출 후 `grid`를 원소 단위 비교 |
| NFR-05(선택) | 단일 스레드 기준, TQ-01·TQ-NOSOL 각각에 대한 end-to-end 처리 시간이 **50ms 미만**이다 | 벽시계 타이머로 100회 중앙값 또는 평균이 `< 50` ms |

**금지(개발 원칙 — Report/3·`magicsquare-forbidden.mdc`와 정합):** 정답 분기를 의미 없는 리터럴 나열로 고정하는 것, 의미 상수를 숫자로만 흩어 쓰는 것(34 등은 **이름 있는 도메인 상수**로 표현), `print` 기반 판정, 테스트 없이 공개 계약 변경.

# 8. Dual-Track TDD Strategy

## 8.1 Track A — Boundary(UI) TDD

**정의:** 여기서 “UI”는 **그래픽 인터페이스가 아니라** 외부 호출자와 Domain 사이의 **경계 계층**(입력 수용·검증·오류 매핑·성공 응답 포장)이다.

**Contract-first 테스트 항목(필수):**

| TC-ID | 조건 | 기대 |
|--------|------|------|
| U-01 | 3×4 등 I1 위반 | `INVALID_SIZE`, Domain 해 결정 **0회** |
| U-02 | `grid==null` 또는 임의 행 `null` | `INVALID_SIZE`, 해 결정 **0회** |
| U-03 | 0이 1개 | `INVALID_ZERO_COUNT` |
| U-04 | 0이 3개 | `INVALID_ZERO_COUNT` |
| U-05 | 17 등 I2 위반 | `INVALID_VALUE_RANGE` |
| U-06 | 1~16 중복 | `DUPLICATE_VALUES` |
| U-07 | TQ-01 | 길이 6의 성공 배열, Domain은 실제 구현 또는 테스트 더블 중 팀이 선택한 방식으로 검증 |
| U-08 | U-07 확장 | 모든 좌표 ∈ [1,4], `n1≠n2`, `n1,n2∈[1,16]` |

**실패 정책 — `code` 및 `message` (고정 문자열)**

| code | message |
|------|---------|
| `INVALID_SIZE` | `Grid must be 4x4.` |
| `INVALID_VALUE_RANGE` | `Each cell must be 0 or 1..16.` |
| `INVALID_ZERO_COUNT` | `There must be exactly two zeros (empty cells).` |
| `DUPLICATE_VALUES` | `Values 1..16 must not duplicate (excluding zeros).` |

## 8.2 Track B — Domain(Logic) TDD

**메서드(개념) 단위 테스트 목록(필수):**

| TC-ID | 대상 | 기대 요약 |

| D-04~D-05 | 0 개수 | 전제 위반 시 FR-01과 동일 분류를 **Domain 검증 함수** 테스트로도 보호(팀이 검증을 Domain에 둔 경우) |

## 8.3 병렬 진행 규칙

1. **동시 RED:** Track A는 U-01, Track B는 D-14 등 **서로 다른 실패 이유**로 테스트를 추가해 동시에 실패 상태를 만든다.  
2. **동시 GREEN:** Boundary의 최소 매핑과 Domain의 최소 판정을 **같은 스프린트 단위**에서 통과시킨다.  
3. **REFACTOR:** 공통 상수(34 등)·중복 검증 로직을 정리하되, **계약 문자열·입출력 스키마**는 회귀 테스트를 통과해야 변경 가능하다.  
4. **금지:** “Domain 전체 완성 후 Boundary 추가” 순서-only 진행.

# 10. Architecture Overview (High-Level)

## 10.1 레이어

| 레이어 | 책임 | 금지 |
|--------|------|------|
| **Boundary** | FR-01, 외부 `int[][]` 수용, 오류 코드·메시지 표준, 성공 시 `int[6]` 전달, Domain 실패 매핑 | 도메인 불변조건을 우회하는 “부분 성공” 응답 |
| **Domain** | FR-02~FR-05, I1~I6 판정, 임시 완전 격자 구성(필요 시), 입력 비변형 | Boundary·파일·DB에 대한 의존 |

## 10.2 SRP / OCP

- **SRP:** 빈칸 탐색, 누락 수 도출, 완성 판정, 순서 정책은 **별 컴포넌트**로 문서화한다(Report/2 §1.1 이름 참조: `EmptyCellLocator`, `MissingNumbersResolver`, `MagicSquareValidator`, `AnswerOrderingPolicy` 등).  
- **OCP:** N×N 확장은 본 PRD Out-of-Scope이며, 향후 확장 시 **새 타입·새 검증기** 추가로 기존 4×4 계약 테스트를 깨지 않는 방향만 허용한다.

## 10.3 의존성 방향

`Boundary → Domain` 단방향. **Domain은 Boundary 타입을 import하지 않는다.**
