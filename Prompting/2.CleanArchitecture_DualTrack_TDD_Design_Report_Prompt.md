---

# 1) Logic Layer (Domain Layer) 설계

## 1.1 도메인 개념

| 구분 | 이름 | 책임(SRP) |
|------|------|-----------|
| Value Object | `Grid4x4` | 4×4 정수 격자, 좌표·셀 접근, 0=빈칸 의미 보존 |
| Value Object | `CellPosition` | (row,col) 1-index 또는 내부 0-index 중 하나로 **일관** 표현; 변환 책임만 |
| Value Object | `MissingPair` | 누락된 두 숫자 집합 {a,b} (순서 없음) 보관 |
| Value Object | `PlacementProposal` | (pos1→nSmall?, pos2→nLarge?) 등 **후보 배치** 표현 |
| Entity | `PartialMagicSquare` (또는 동등 개념) | 현재 격자 + 빈칸 2곳 + 채워진 값 집합에 대한 **불변조건 검증** 진입점 |
| Domain Service | `MagicSquareValidator` | 행/열/두 대각선 합=34(아래 불변) 판정 |
| Domain Service | `EmptyCellLocator` | 빈칸(0) 정확히 2개 위치 나열(순서 규칙은 유스케이스에서 고정) |
| Domain Service | `MissingNumbersResolver` | 1~16 중 미등장 두 수 도출 |
| Domain Service | `AnswerOrderingPolicy` | 출력 계약 `[r1,c1,n1,r2,c2,n2]`의 (n1,n2) 순서 결정: (작→첫빈, 큰→둘째빈) 우선 시도 후 실패 시 반대 |

**의존 방향**: Entity/VO는 서로·Validator에만 의존하고, **UI/Data에 의존하지 않음**.

## 1.2 도메인 불변조건(Invariants)

| ID | 불변조건 | 검증 가능 규칙(측정 가능) |
|----|-----------|---------------------------|
| I1 | 격자 크기는 **4×4** | 행 수=4, 각 행 길이=4 |
| I2 | 값은 **0 또는 1~16** | 각 셀 ∈ {0}∪[1,16] |
| I3 | **0은 정확히 2개** | count(0)=2 |
| I4 | **1~16은 각 최대 1회** (0 제외 중복 없음) | ∀v∈[1,16], count(v)≤1 |
| I5 | **마방진 완성 시** 각 행 합=34, 각 열 합=34, 주대각·부대각 합=34 | 4행+4열+2대각 = 10개 합 모두 34 |
| I6 | **Magic constant** (이 문제의 합 목표)는 **34** (1~16 표준 4×4) | I5와 동일 목표값으로 고정 |

**주의**: “부분 입력” 단계에서는 I5가 깨질 수 있음. 도메인은 **(a) 구조 불변 I1~I4**와 **(b) 완성 판정 I5**를 **별 연산**으로 분리해 테스트해야 모호함이 사라짐.

## 1.3 핵심 유스케이스(도메인 관점)

| UC | 입력(도메인 내부 표현) | 출력 | 실패 |
|----|-------------------------|------|------|
| UC-A | `Grid4x4` | 빈칸 두 좌표 `(r1,c1),(r2,c2)` (1-index, **고정 순서 규칙**) | 빈칸 개수≠2 |
| UC-B | `Grid4x4` | `MissingPair {a,b}` | 누락 수 개수≠2 (중복/범위 위반 시) |
| UC-C | 완성된(0 없음) `Grid4x4` | boolean + 위반 목록(선택) | - |
| UC-D | 두 빈칸 위치 + 두 수 배치 후보 2가지 | **성공한 배치** 하나 선택 | 둘 다 실패 시 “해 없음” |

**빈칸 순서 고정 규칙(검증 가능)**:  
- 행 우선 스캔(row 1→4, col 1→4)에서 **먼저 만나는 0**이 “첫 빈칸”, **나중 0**이 “둘째 빈칸”.  
- 출력 `(r1,c1)`은 첫 빈칸, `(r2,c2)`는 둘째 빈칸.

**숫자 순서 규칙(문제에서 주어진 계약과 정합)**:  
- `n_small = min(a,b)`, `n_large = max(a,b)`.  
- 시도1: 첫 빈칸←`n_small`, 둘째 빈칸←`n_large`가 I5 만족하면 `(n1,n2)=(n_small,n_large)`.  
- 아니면 시도2: 반대 배치가 I5 만족하면 `(n1,n2)=(n_large,n_small)`.  
- 둘 다 불만족이면 도메인 실패(해 없음).

## 1.4 Domain API(내부 계약)

**코드 없이 시그니처 수준(의사형식)**:

1. `parseAndValidateStructure(grid:int[][]) -> Result<Grid4x4, DomainError>`  
   - **입력**: 외부와 동일한 4×4 int[][]  
   - **출력**: 구조적으로 유효한 VO  
   - **실패조건**: I1~I4 위반(크기, 범위, 0 개수, 중복)

2. `locateTwoZeros(grid:Grid4x4) -> Result<((r1,c1),(r2,c2)), DomainError>`  
   - **출력**: 1-index, 스캔 순서 고정  
   - **실패조건**: 0이 2개가 아님(이미 parse에서 걸러지면 중복 방지 가능)

3. `findMissingTwoNumbers(grid:Grid4x4) -> Result<MissingPair, DomainError>`  
   - **출력**: {a,b}  
   - **실패조건**: 1~16 등장 개수 합이 14가 아님(즉 누락이 2개가 아님) — 구조 불변과 연동

4. `isCompletedMagicSquare(grid:Grid4x4) -> boolean`  
   - **전제(호출 계약)**: 0 없음(위반 시 `false` 또는 별도 실패 타입 중 하나로 **문서 고정**)

5. `tryFillAndDecideOrder(grid:Grid4x4, holes, pair) -> Result<[r1,c1,n1,r2,c2,n2], NoSolutionError>`  
   - **출력**: 외부 출력 계약과 동일 int[6]  
   - **실패조건**: 두 배치 모두 I5 불만족

**실패 타입(검증 가능 분류)**:

| DomainError 코드 | 의미 |
|------------------|------|
| `SIZE` | 4×4 아님 |
| `RANGE` | 0/1~16 밖 값 |
| `ZERO_COUNT` | 0 개수 ≠2 |
| `DUPLICATE` | 1~16 중복 |
| `NO_MAGIC_SOLUTION` | 두 배치 모두 실패 |

## 1.5 Domain 단위 테스트 설계(RED 우선)

### 테스트 케이스 목록

| TC-ID | 유형 | 입력 요약 | 기대 | 보호 invariant |
|-------|------|-----------|------|----------------|
| D-01 | 정상 | 알려진 해 1개(문헌/생성된 완성 격자에서 2칸을 0으로 바꾸고 누락 수도 일치) | int[6]가 **재구성 시 I5 만족** | I5 |
| D-02 | 정상 | D-01의 반대 배치가 틀리도록 설계된 케이스 | (n1,n2)가 규칙대로 선택됨 | 출력 순서 규칙 |
| D-03 | 정상 | 두 배치 모두 가능한 입력이 **존재하지 않도록** 문제 특성상 배제 가능하면 생략, 존재하면 “유일 규칙” 테스트로 고정 | 규칙대로 1개만 성공 | UC-D |
| D-04 | 비정상 | 0이 1개 | `ZERO_COUNT` | I3 |
| D-05 | 비정상 | 0이 3개 | `ZERO_COUNT` | I3 |
| D-06 | 비정상 | 값 17 포함 | `RANGE` | I2 |

| D-08 | 비정상 | 1~16 중복 | `DUPLICATE` | I4 |
| D-09 | 비정상 | 3×3/4×5 등 크기 | `SIZE` | I1 |
| D-10 | 비정상 | 행 길이 불일치(가변 행) | `SIZE` | I1 |
| D-11 | 엣지 | “거의 맞지만 합 34 실패”로 두 배치 모두 실패 | `NO_MAGIC_SOLUTION` | I5 |
| D-12 | 엣지 | 빈칸 좌표 스캔 순서: 특정 패턴으로 (r1,c1)이 먼저임을 고정 검증 | 좌표 일치 | 빈칸 순서 규칙 |

| D-14 | 단위 | 완성 마방진 | `true` | I5 |

**RED 우선 순서(권장)**:

4. 최종 조합/순서(D-01~D-03)  
5. 구조 오류 매핑(D-04~D-10)

# 2) Screen Layer (UI Layer) 설계 (Boundary Layer)

## 2.1 사용자/호출자 관점 시나리오

| 단계 | 행위 | 결과 |
|------|------|------|
| S1 | 호출자가 4×4 int[][] 제공 | Boundary가 raw 입력 수용 |
| S2 | Boundary가 스키마/구조 검증 | 통과 시 Domain 호출 준비 |
| S3 | Domain에 위임 | 성공 시 int[6], 실패 시 도메인 오류 코드 |
| S4 | Boundary가 출력 스키마로 변환 | int[6] 또는 표준 에러 객체 |

## 2.2 UI 계약(외부 계약)

### Input schema (고정)

| 필드 | 타입 | 규칙 |
|------|------|------|
| `grid` | int[][] | 길이=4, 각 행 길이=4 |
| `grid[r][c]` | int | 0 또는 1~16 |
| zeros | derived | 정확히 2 |
| values 1..16 | derived | 중복 없음 |

### Output schema (성공, 고정)

| 필드 | 타입 | 규칙 |
|------|------|------|
| `answer` | int[6] | `[r1,c1,n1,r2,c2,n2]` |
| 좌표 | int | 1-index, 범위 1~4 |
| `n1,n2` | int | 누락 두 수, 순서는 문제에서 정의한 규칙 |

### Error schema (고정)

| code | meaning | HTTP 비유 없음(로컬 API 가정) |
|------|---------|-------------------------------|
| `INVALID_SIZE` | 4×4 아님 | I1 |
| `INVALID_VALUE_RANGE` | 값 범위 위반 | I2 |
| `INVALID_ZERO_COUNT` | 0 개수 ≠2 | I3 |
| `DUPLICATE_VALUES` | 1~16 중복 | I4 |
| `NO_SOLUTION` | 도메인에서 해 없음 | `NO_MAGIC_SOLUTION` |

**Boundary의 책임 분리(검증 가능)**:

- 크기/타입/null 여부는 Boundary에서 먼저 실패 처리 가능  
- “중복/0개수/범위”는 **Domain과 중복 검증**을 피하려면:  
  - **정책 A(추천)**: Boundary는 최소(널/길이)만, 나머지는 Domain 단일 진실원천  
  - **정책 B**: Boundary에서 전부 검증하고 Domain은 “내부 불가능 상태 없음” 가정  
  - 문서에 A 또는 B 중 **하나만 선택**하고 테스트가 그 선택을 강제

## 2.3 UI 레벨 테스트(Contract-first, RED 우선)

**전제**: Domain 포트를 Mock/Stub으로 주입해 Boundary만 검증.

| UI-TC | 내용 | 기대 | Mock 설정 |
|-------|------|------|------------|
| U-01 | 잘못된 크기(3×4) | `INVALID_SIZE` | Domain 호출 0회 |
| U-02 | null grid / null row | `INVALID_SIZE` 또는 `INVALID_SIZE`로 통일 | 호출 0회 |
| U-03 | 빈칸 1개 | `INVALID_ZERO_COUNT` | 호출 0회(정책 A면 Domain) |
| U-04 | 빈칸 3개 | `INVALID_ZERO_COUNT` | 동일 |
| U-05 | 값 99 | `INVALID_VALUE_RANGE` | 동일 |
| U-06 | 중복 숫자 | `DUPLICATE_VALUES` | 동일 |
| U-07 | 정상 입력 | Domain이 int[6] 반환 | Mock 고정 반환 |
| U-08 | 반환 포맷 검증 | 길이 6, 좌표 1~4, n1 n2가 1~16, 서로 다름 | U-07 연장 |
| U-09 | Domain이 NO_SOLUTION | `NO_SOLUTION`로 매핑 | Mock 실패 |
| U-10 | Domain이 구조 오류 반환 시(정책 A) | Boundary가 그대로 전달/매핑 규칙 일치 | Mock 에러 |

**Contract-first 체크리스트**

- [ ] 성공 응답에 **항상** `answer:int[6]` 키 존재(또는 언어 관례상 동등)  
- [ ] 실패 응답에 **항상** `code` + `message` 존재(문구 규칙은 2.4)  
- [ ] Domain 호출 횟수가 시나리오별로 **정확히 N회**인지 검증(U-01 등)

## 2.4 UX/출력 규칙

**에러 메시지 표준(문구 고정 — 구현 시 그대로 사용)**

| code | message (정확 문자열) |
|------|------------------------|
| `INVALID_SIZE` | `Grid must be 4x4.` |
| `INVALID_VALUE_RANGE` | `Each cell must be 0 or 1..16.` |
| `INVALID_ZERO_COUNT` | `There must be exactly two zeros (empty cells).` |
| `DUPLICATE_VALUES` | `Values 1..16 must not duplicate (excluding zeros).` |
| `NO_SOLUTION` | `No placement makes a 4x4 magic square with magic sum 34.` |

**성공 출력(텍스트 변환이 있다면 고정 예시)**  
- 한 줄: `answer: [r1,c1,n1,r2,c2,n2]` 형식으로 **쉼표+공백** 구분을 문서에 고정.

# 3) Data Layer 설계 (Data Layer)

## 3.1 목적 정의

| 항목 | 내용 |
|------|------|
| 필요성 | 학습 목적: **동일 유스케이스**에 대해 저장소 교체(메모리/파일)로 **테스트/실행 재현성** 확보 |
| 범위 | “문제 입력”과 “마지막 성공 결과” 정도만 저장하면 됨(DB/마이그레이션 없음) |

## 3.2 인터페이스 계약

**의사 API(코드 없음)**

1. `MatrixRepository.saveSession(inputGrid, outputAnswerOptional)`  
   - **저장 대상**: 입력 행렬(4×4), 선택적으로 결과 int[6]  
   - **실패**: 쓰기 실패(파일), 직렬화 불가 타입

2. `MatrixRepository.loadLastSession() -> Optional<SessionRecord>`  
   - **SessionRecord 필드(고정)**: `grid:int[][]`, `answer:int[6]?`, `savedAt:instant`

3. `MatrixRepository.clear()` (선택)  
   - 테스트 초기화용

## 3.3 구현 옵션 비교

| 옵션 | 장점 | 단점 | 학습 적합 |
|------|------|------|-----------|
| A InMemory | 테스트 빠름, 파일 의존 0 | 프로세스 종료 시 소실 | 단위/통합 최적 |
| B File(JSON) | 사람이 읽기 쉬움, 수동 검증 쉬움 | IO flake, 경로 권한 | “교체 가능” 입증에 유리 |
| B' File(CSV) | 간단 | 2차원 표현이 번거로움 | 비추(규칙 검증 비용↑) |

**추천안**: **옵션 A를 기본 + 옵션 B(JSON)를 스위치 가능한 2구현**  
**이유**: 통합 테스트는 InMemory로 안정적으로 회귀 보호하고, 파일 저장은 별도 어댑터 테스트로 IO 실패를 격리할 수 있음.

## 3.4 Data 레이어 테스트

| DT | 시나리오 | 기대 |
|----|-----------|------|
| DT-01 | save 후 load | grid deep-equal, answer 동일 |
| DT-02 | answer 없이 save | load 시 answer null/empty 정책 고정 |
| DT-03 | 파일 없음(load) | 명시적 예외 코드 `NOT_FOUND` (문구 고정 가능) |
| DT-04 | JSON 깨짐 | `CORRUPT_FILE` |
| DT-05 | 불변조건: 로드된 grid가 4×4 아니면 거부 | `INVALID_PERSISTED_DATA` |
| DT-06 | 값 범위 깨짐 | 동일 |

# 4) Integration & Verification (통합 및 검증)

## 4.1 통합 경로 정의

**의존성 방향(안쪽으로만)**:

`Boundary(UI)` → `Application(선택: Orchestrator)` → `Domain`  
`Application` → `Data` (인터페이스)  
`Domain` **은** `Data/UI`에 의존하지 않음.

| 경로 | 설명 |
|------|------|
| R1 | Boundary 검증 → Domain.solve → Boundary 출력 |
| R2 | Boundary 검증 → Application.solveAndPersist → Domain → Repository.save |
| R3 | Application.loadLast → Boundary 출력 포맷(조회 기능이 있을 때만) |

## 4.2 통합 테스트 시나리오

### 정상(2개 이상)

| IT | 단계 | 검증 |
|----|------|------|
