프로젝트는 이미 **`main`** 기준으로 GitHub에 올라가 있고, 규칙상 TDD는 **Red → Green → Refactor**만 맞추면 됩니다. Dual-Track(UI 트랙 + Logic 트랙)과 맞춘 **브랜치 전략**을 아래처럼 쓰면 정리하기 좋습니다.

---

## 1. 기본 구조 (장수 브랜치)

| 브랜치 | 역할 |
|--------|------|
| **`main`** | 항상 **테스트 통과·커버리지 기준**을 만족하는 통합본. 배포·데모·PR 머지 대상. |
| **`develop`** (선택) | 여러 기능을 모아 통합 테스트하기 전 버퍼. 팀이 작거나 혼자면 **`main`만** 써도 됨. |

혼자/소규모면 **`main` + 짧은 기능 브랜치**만으로 충분합니다.

---

## 2. 작업 브랜치 네이밍 (Dual-Track + 단계 반영)

기능 단위(에픽/시나리오/PRD 섹션)마다 브랜치를 두고, **같은 브랜치 안에서** Red → Green → Refactor를 돌립니다.

**권장 패턴**

```text
feat/<scope>-<short-slug>
```

예:

- `feat/4x4-validate-rows` — 로직 트랙
- `feat/ui-grid-input` — UI 트랙  
- `feat/control-place-tile` — ECB Control 쪽

**트랙을 이름에 넣고 싶다면** (선택):

```text
feat/logic-magic-constant
feat/ui-board-render
```

한 PR에 UI+로직을 같이 넣을지, PR을 쪼갤지는 팀 속도에 맞추되, **커밋 메시지로 단계를 드러내는 방식**이 Dual-Track TDD와 잘 맞습니다.

---

## 3. 커밋과 TDD 단계 정렬 (당신이 말한 순서)

진행 순서를 브랜치가 아니라 **커밋 경계**로 나누는 것을 권장합니다.

1. **Dual-Track UI + Logic TDD**  
   - UI 쪽 실패 테스트 + Logic 쪽 실패 테스트를 **먼저** 추가(의도된 Red).  
   - 커밋 예: `test: red — dual-track for <behavior>`  
   - 필요하면 **UI 테스트 커밋**과 **logic 테스트 커밋**을 나눠도 됨.

2. **GREEN**  
   - 통과에 필요한 **최소** 구현만.  
   - 커밋 예: `feat: green — minimal impl for <behavior>`  
   - 리팩터링은 이 커밋에 넣지 않음.

3. **Dual-Track Refactoring**  
   - 동작 유지, ECB·이름·중복 정리, 커버리지 80% 이상 유지.  
   - 커밋 예: `refactor: dual-track cleanup for <behavior>`

이렇게 하면 PR 리뷰 시 **Red / Green / Refactor**가 `git log`만으로도 보입니다.

---

## 4. PR·머지 흐름

- **`feat/*` → `main`** (또는 `feat/*` → `develop` → `main`).  
- PR 제목에 범위를 적기: `[4x4] Dual-track: row sum validation`.  
- 머지 전: `pytest`, 커버리지, 포매터/린트(프로젝트에 있으면) 통과.  
- 머지 방식:  
  - **Squash merge**: PR 하나 = 기능 단위로 히스토리 단순화.  
  - **Merge commit**: Red/Green/Refactor 커밋을 `main`에 그대로 남기고 싶을 때.

---

## 5. Dual-Track을 브랜치로 쪼개는 경우 (선택)

UI와 로직을 **완전히 다른 사람/스프린트**로 나눌 때만:

- `feat/ui/...` 와 `feat/logic/...` 를 따로 두고,  
- **통합 지점**에서만 `main`에 순서대로 머지(로직이 Boundary/API를 먼저 정의하면 UI가 그 계약에 맞춤).

대부분의 Dual-Track TDD는 **한 기능 브랜치 + 두 트랙 테스트가 같은 PR**이 협업 비용이 적습니다.

---

## 6. 요약

- **브랜치**: `main` 고정 + `feat/<영역>-<기능>` 짧은 수명 브랜치.  
- **TDD 단계**: 브랜치를 세 단계로 쪼개기보다, **한 브랜치에서 커밋 3덩어리**(Red 듀얼 트랙 → Green → Refactor)로 맞추기.  
- **이름**: scope(예: `4x4`, `control`, `ui`)를 넣어 나중에 PRD·에픽과 대응시키기 쉽게 유지.

원하시면 PRD의 에픽 ID(예: `US-01`)를 브랜치명에 붙이는 규칙까지 맞춘 **한 줄 템플릿**도 정리해 드리겠습니다.
