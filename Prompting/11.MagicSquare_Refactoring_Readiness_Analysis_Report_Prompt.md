### 1. `test_*.py` 존재 여부와, 부족하면 어디에 테스트가 필요한지

**존재합니다.** 예를 들면 다음과 같습니다.

| 경로 | 비고 |
|------|------|
| `tests/magicsquare/test_magic_square_logic_red.py` | `magicsquare.domain`의 `find_blank_coords` 등을 import |
| `tests/magicsquare/test_magic_square_ui_boundary_red.py` | `magicsquare.boundary`의 `validate` 등을 import |
| `tests/entity/models/test_user.py` | 엔티티 쪽 |

다만 **말씀하신 파일별로 보면**:

- **`domain.py`**: 위 logic RED 테스트에 연결되어 있지만, 대부분 `pytest.fail("RED")` 스켈레톤이고, 실제로 의미 있는 assertion이 도는 케이스는 일부(예: `find_blank_coords` 한 개) 수준입니다. **도메인 전체 동작을 덮는 테스트는 아직 부족**합니다.
- **`boundary.py`**: boundary RED 테스트에 연결되어 있고, 4×4가 아닐 때 `validate`가 `ValueError`를 내는 정도는 검증하지만, 나머지 시나리오도 대부분 RED 스켈레톤입니다. **경계 검증 규칙을 끝까지 검증하는 테스트는 부족**합니다.
- **`gui/main_window.py`**: `tests` 아래에서 `main_window`, `MainWindow`, `gui` 관련 import/언급이 **없습니다**. 따라서 **이 파일(및 GUI에서 조립·이벤트 처리하는 부분)에 대한 전용 테스트가 사실상 없다**고 보는 것이 맞고, 리팩토링 전에 **여기에 대한 테스트(또는 최소한 추출 가능한 순수 로직에 대한 테스트)**가 특히 필요합니다.

### 1. 각 파일이 담당하는 ECB 역할

**`domain.py` → Entity(데이터) vs Control(로직)?**  
- 이 파일에는 **상태를 들고 있는 Entity 클래스나 전용 데이터 구조가 없고**, `list[list[int]]` 그리드에 대한 **순수 함수(규칙·판정·풀이 시도)**만 있습니다.  
- 사용자가 나눈 기준(데이터 보관 vs 비즈니스 규칙 실행)으로 보면 **주 역할은 Control(비즈니스 규칙·도메인 연산 실행)**에 가깝고, “데이터 보관”은 이 레이어의 책임이 아니라 **호출자가 넘기는 값**에 가깝습니다.

**`boundary.py` → Boundary를 제대로 하고 있는가?**  
- PyQt 없이, **외부에서 들어온 그리드**에 대해 `ValueError`와 고정 메시지로 **입력 규칙(FR/BR)을 적용**하고, 도메인의 `solve_placement`를 호출해 **결과를 반환**합니다.  
- ECB에서 Boundary는 “액터(사용자·외부)와 시스템 사이의 접착”인데, **`main_window`가 UI Boundary**, **`boundary.py`는 그 UI와 도메인 사이의 응용 경계(검증·오케스트레이션·예외 매핑)**로 동작하고 있어 **Boundary 역할에 부합**합니다. (이름이 `boundary`인 것과 역할도 대체로 일치합니다.)

**`gui/main_window.py` → UI 외 책임이 섞였는가?**  
- 위젯 구성, 그리드 읽기, 버튼 처리, 메시지 박스·결과 표시 등 **표현·입출력**이 대부분입니다.  
- `validate` / `solve` 호출은 **유스케이스를 경계 모듈에 위임**하는 것으로 ECB상 자연스럽고, **도메인 규칙을 GUI 안에 풀어 쓰지는 않습니다.**  
- 다만 **같은 입력에 대해 검증이 경계에서 두 번 일어날 수 있는 호출 순서**(GUI에서 `validate` 후 `solve`가 내부에서 다시 `validate`)는 **레이어 혼입이라기보다 중복 호출**에 가깝습니다.

### 2. 잘못된 위치에 있는 코드 — 옮겨야 할 곳

- **레이어 위반이라고 보기 어려운 코드는 없습니다.** (도메인에 UI 없음, GUI에 도메인 규칙 없음, 경계에서 검증·위임 구조가 명확함.)  
- 정리만 하자면, **“검증 + 풀기”를 한 번에 쓰고 싶다면** GUI는 **`solve`만 호출**해도 되므로, 그때 줄어드는 것은 **호출 패턴**이지 “다른 파일로 옮겨야 하는 로직 덩어리”는 아닙니다.

---

### 3. `domain.py` 안 Entity vs Control 혼재 — 분리가 논의될 만한 부분

명시적 Entity가 없어도, **역할만** 나누면 다음이 **Control 쪽(유스케이스 조율)**에 가깝고, 나머지는 **그리드에 대한 도메인 규칙/질의**에 가깝습니다.

| 구분에 가까운 부분 | 내용 |
|-------------------|------|
| **Control에 가깝다** | `solve_placement`: 빈 칸·빠진 수를 구한 뒤 **두 배치를 순서대로 시도**하고, 실패 시 예외를 올리는 **흐름·정책(FR-05)** 조율. |
| **Entity/도메인 규칙(질의·불변식)에 가깝다** | `find_blank_coords`, `find_missing_pair_sorted`, `is_completed_magic_square`, `_copy_grid`: 주로 **한 그리드에 대한 판정·추출·복사** 같은 **규칙·상태 판별**. |

**분리가 “필수”인지**: ECB를 **클래스 단위로 엄격히** 나누는 팀이라면, 위 표처럼 **“배치 시도”(Control)**와 **“마방진 판정·좌표·빠진 수”(Entity 또는 도메인 서비스)**를 파일/모듈 또는 타입으로 나누는 논의가 가능합니다. 지금은 **한 모듈에 함수로 응집**되어 있어 동작상 문제는 없고, **구조적 분리는 취향·규모·테스트 전략**에 가까운 선택입니다.

---

**한 줄 요약**: `domain.py`는 데이터 보관보다 **규칙·연산(Control 성격)**이 중심이고, `boundary.py`는 **경계(검증·위임·예외 변환)**로 적절하며, GUI는 **UI + 경계 API 호출**만 하고 도메인은 섞이지 않습니다; `domain.py` 내부에서만 보면 **`solve_placement`가 나머지 함수들에 비해 “유스케이스 조율” 성격이 강해** Entity/Control을 나눌 때 가장 먼저 쪼개기 쉬운 경계입니다.

## 테스트 선행 필요 항목

리팩토링 전에 **동작을 고정**할 테스트를 두는 것이 안전합니다. 현재 `tests/magicsquare/*_red.py`에는 `pytest.fail("RED")`가 많아, 아래 함수들에 대해 **실제 assertion이 있는 GREEN 테스트**를 먼저 채우는 것을 권장합니다.

- **`boundary.validate`** — `None`/행 수·열 길이, 값 범위, 0이 정확히 2개가 아닌 경우, 중복(0 제외) 등 **메시지 문자열(`MSG_*`)까지** 기대값 고정.
- **`boundary.solve`** — 유효 입력에서 `solve_placement`와 동일한 6원소 반환, `NoMagicSolutionError` 경로에서 `ValueError(MSG_NO_SOLUTION)` 및 `__cause__` 확인.
- **`domain.solve_placement`** — 알려진 그리드(또는 PRD TQ 케이스)에 대해 성공 시 `[r1,c1,n1,r2,c2,n2]`, 실패 시 `NoMagicSolutionError`.
- **`domain.is_completed_magic_square`** — 완성 마방진 True, 0이 남은 경우 False, 행/열/대각만 맞는 반례 False 등.
- **`domain.find_blank_coords`**, **`domain.find_missing_pair_sorted`** — 이미 일부 테스트가 있으면 나머지 RED 슬롯을 GREEN으로 채움.
- **`gui/main_window.py`** (선택) — PyQt는 단위 테스트 비용이 크므로, **최소한** 위 도메인·경계가 충분히 덮이면 수동 스모크로도 가능; 자동화한다면 `pytest-qt` 등으로 “풀기 클릭 시 결과 라벨 갱신”만 스팁.

### 회귀 테스트(Regression Test) 실행 명령어

프로젝트 루트(`pyproject.toml`이 있는 디렉터리)에서:

```bash
pytest
```

전체가 아니라 마방진 모듈만 돌릴 때:

```bash
pytest tests/magicsquare
```

### 외부 동작(기능)이 바뀌지 않았음을 확인하는 방법

1. **자동**: 위 `pytest`가 **전부 통과**하고, 특히 `validate` / `solve` / `solve_placement` / `is_completed_magic_square`에 대한 **기존에 고정해 둔 입출력·예외 메시지** assertion이 그대로 통과하는지 확인합니다.  
2. **수동(GUI)**: 앱 실행 후 동일 입력에 대해 **이전과 같은 경고 문구**(예: `MSG_*`에 해당하는 한글/영문 메시지가 경계에서 그대로 나오는지), 풀기 성공 시 **결과 라벨 형식**(쉼표 구분 6개 숫자)이 동일한지 몇 가지 대표 케이스(유효 입력, 무해 입력, 해 없음)로 확인합니다.  
3. **비기능 회귀**: 리팩토링이 “검증 한 번만” 같은 내부 최적화라면, 사용자에게 보이는 성공/실패 조건은 변하지 않아야 하므로, **테스트에 고정한 공개 메시지·예외 타입**이 바뀌지 않았는지 diff로 검토합니다.

## 3. ECB(Entity–Control–Boundary) 분석 요약

| 파일 | ECB 역할 요약 |
|------|----------------|
| `domain.py` | **상태를 들인 Entity 클래스는 없고**, `list[list[int]]`에 대한 **순수 도메인 연산(규칙 실행)**이 중심이다. 사용자 구분(데이터 보관 vs 규칙 실행)으로는 **Control(비즈니스 규칙·연산 실행)**에 가깝다. |
| `boundary.py` | 외부 입력에 `ValueError`·고정 메시지로 검증하고 도메인을 호출한다. **UI와 도메인 사이의 응용 경계(검증·오케스트레이션·예외 매핑)**로 Boundary 역할에 부합한다. |
| `gui/main_window.py` | 위젯·이벤트·표현이 주이며 도메인 규칙을 풀어 쓰지 않는다. **이중 `validate` 호출**은 레이어 혼입이라기보다 **중복 호출**에 가깝다. |

**잘못된 레이어 배치:** 명백한 위반은 없다(도메인에 Qt 없음, GUI에 마방진 판정 없음).

**`domain.py` 내부 Entity vs Control이 논의될 만한 경계:** `solve_placement`는 빈 칸·빠진 수 조회 후 두 배치를 시도하는 **유스케이스 조율(Control 성격)**에 가깝고, `find_blank_coords`, `find_missing_pair_sorted`, `is_completed_magic_square`, `_copy_grid`는 **그리드에 대한 규칙·질의**에 가깝다. 클래스 단위 ECB를 엄격히 적용할 경우 **`solve_placement`를 먼저 분리하기 쉬운 경계**로 볼 수 있다.
