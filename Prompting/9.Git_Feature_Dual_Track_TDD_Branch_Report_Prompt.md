**User**

당신은 Dual-Track UI + Logic TDD 전문가입니다.

우리는 이미 Magic Square(4x4) 시나리오를 작성했습니다.
지금 단계는 구현이 아니라,
시나리오를 “실패하는 테스트(RED)”로 변환하는 것입니다.

중요
- 구현 코드를 작성하지 마십시오.
- GREEN 단계로 가지 마십시오.
- REFACTOR 단계로 가지 마십시오.
- 오직 RED 테스트만 작성하십시오.
- 모든 테스트는 현재 구현이 없기 때문에 실패해야 합니다.

------------------------------------------------------------
프로젝트 조건
------------------------------------------------------------

입력:
- 4x4 int[][]
- 0은 빈칸
- 정확히 2개의 0
- 값 범위 1~16
- 중복 금지 (0 제외)

출력:
- int[6]
- 좌표는 1-index
- [r1, c1, n1, r2, c2, n2]

마방진 상수:
- 4x4 → 34

------------------------------------------------------------
Dual-Track RED 단계 구조
------------------------------------------------------------

UI RED        Logic RED
   ↓              ↓
(둘 다 실패 상태)

------------------------------------------------------------
TRACK A — UI / Boundary RED
------------------------------------------------------------

다음을 실패 테스트로 변환하십시오:

1. 4x4가 아닌 입력은 예외 발생
2. 빈칸이 2개가 아니면 예외 발생
3. 값 범위 위반 시 예외 발생
4. 중복 숫자 존재 시 예외 발생
5. 반환 배열 길이는 6이어야 한다
6. 반환 좌표는 1-index여야 한다

각 테스트에 대해:
- 테스트 이름
- Given (입력)
- When (호출)
- Then (기대)
- 이 테스트가 보호하는 계약 설명

------------------------------------------------------------
TRACK B — Logic RED
------------------------------------------------------------

다음을 실패 테스트로 작성하십시오:

1. find_blank_coords()
   - row-major 순서로 정확히 2개 반환

2. find_not_exist_nums()
   - 누락 숫자 2개 반환
   - 오름차순

3. is_magic_square()
   - 모든 행 합 동일
   - 모든 열 합 동일
   - 두 대각선 합 동일
   - 34와 일치

4. solution()
   - small → first blank 시도
   - 실패 시 reverse 시도
   - 6-length 반환
   - 1-index 좌표

각 테스트는:
- 실패 상태임을 명확히 설명
- 어떤 Invariant를 검증하는지 명시

# UI RED Tests

| Test ID | Scenario | Invariant 보호 내용 |
|--------|----------|---------------------|
| **UI-RED-01** | **Given** 3×3 또는 5×5 등 4×4가 아닌 `int[][]`가 주어짐. **When** 검증/솔루션 진입점(예: `validate_grid` 또는 `solution`의 최상위)을 호출함. **Then** 도메인 예외(또는 프로젝트에서 정한 `ValueError` 등)가 발생해야 함. | **그리드 형상 계약:** 입력은 반드시 4행·4열이어야 하며, 그렇지 않으면 어떤 연산도 수행하지 않는다. |
| **UI-RED-02** | **Given** 4×4이지만 `0`(빈칸)의 개수가 0, 1, 3, … 등 **정확히 2가 아님**. **When** 동일 진입점 호출. **Then** 예외 발생. | **빈칸 개수 계약:** “정확히 두 칸만 비어 있음”이 깨지면 후속 로직(두 좌표·두 숫자)이 정의되지 않으므로 즉시 실패해야 한다. |
| **UI-RED-03** | **Given** 4×4, 빈칸 2개, 그러나 어떤 칸의 값이 **1~16 밖**(0, 17, 음수 등). **When** 호출. **Then** 예외 발생. | **값 도메인 계약:** 0은 빈칸 전용, 나머지는 반드시 1~16에 속해야 한다. |
| **UI-RED-04** | **Given** 4×4, 빈칸 2개, 모든 값이 범위 내이나 **0을 제외한 숫자에 중복**이 있음. **When** 호출. **Then** 예외 발생. | **순열 부분집합 계약:** 1~16 중 이미 채워진 칸은 서로 달라야 하며, 중복이면 “누락 2개” 추론이 모호해진다. |
| **UI-RED-05** | **Given** 유효한 형태의 입력(형상·빈칸·범위·중복 모두 통과 가능한 보드). **When** “답을 돌려주는” API(예: `solution`)를 호출한다고 가정할 때, **아직 구현이 없으면** 호출 자체가 실패하거나, 스텁이 있다면 **반환 길이 검증 단계**에서 실패. **Then** 최종적으로 **길이 6인 `int[]`**만 허용. | **출력 형상 계약:** 답은 항상 `[r1,c1,n1,r2,c2,n2]` 여섯 요소이어야 한다. |
| **UI-RED-06** | **Given** 솔버가 어떤 6-튜플을 반환한다고 가정(또는 픽스처로 고정). **When** 각 `r*`, `c*`가 **1~4** 범위인지 검증. **Then** 0-based 좌표나 5 이상이면 실패. | **출력 좌표 체계 계약:** 좌표는 **1-index**, 행·열 모두 1~4에 한정된다. |

> **RED 상태:** 위 01~04는 “검증 레이어”만 있어도 구현 없이 예외 경로를 먼저 검증할 수 있고, 05~06은 솔버 본체가 없으면 자연스럽게 RED가 유지된다.

# Logic RED Tests

| Test ID | Scenario | Invariant 보호 내용 |
|--------|----------|---------------------|
| **LOG-RED-01** | **`find_blank_coords`** — **Given** 빈칸이 정확히 두 칸인 4×4. **When** `find_blank_coords(grid)` 호출. **Then** 길이 2의 리스트(또는 튜플 쌍 두 개), **행 우선(row-major)** 순서로 앞쪽 행·열이 먼저 오고, 동일 행이면 열 오름차순. (구현 없으면 호출 실패 또는 빈 반환으로 RED.) | **빈칸 순서 불변:** “첫 빈칸 / 두 번째 빈칸”이 전역적으로 정의되어 `solution`의 시도 순서와 일치해야 한다. |
| **LOG-RED-02** | **`find_not_exist_nums`** — **Given** 0 두 개를 제외한 나머지 14칸이 1~16의 서로 다른 값. **When** `find_not_exist_nums(grid)` 호출. **Then** 길이 2, 값은 {1..16} 중 그리드에 없는 두 수, **오름차순** 정렬. | **누락 숫자 집합 불변:** 채워야 할 두 수는 집합적으로 유일하게 결정되며, 표현 순서도 고정되어야 한다. |
| **LOG-RED-03a** | **`is_magic_square`** — **Given** 완전히 채워진 4×4(0 없음), 모든 행 합이 34. **When** `is_magic_square` 호출. **Then** 행 합 일치는 만족하나 **열 또는 대각선**이 깨진 보드 하나로 `False` 기대. | **행 합 마방진 필요조건:** 행만 맞고 마방진이 아닌 경우를 거른다. |
| **LOG-RED-03b** | 동일 API, **열**은 모두 34이나 **행** 중 하나라도 다름 → `False`. | **열·행 대칭 불변:** 마방진 판정은 행·열을 동등하게 본다. |
| **LOG-RED-03c** | 동일 API, 행·열 모두 34이나 **주대각선 또는 부대각선** 하나라도 34 아님 → `False`. | **대각선 불변:** 4×4 마방진 정의에서 두 대각선 합도 마법 상수와 같아야 한다. |
| **LOG-RED-03d** | **Given** 알려진 완성 4×4 마방진 한 개. **When** 호출. **Then** `True`,且 모든 행·열·대각선 합이 **34**와 일치(상수 하드코딩 또는 `MAGIC_SUM`과 동일). | **마법 상수 34:** 문제 도메인의 단일 진리원(`MAGIC_SUM == 34`)과 판정이 일치해야 한다. |
| **LOG-RED-04a** | **`solution`** — **Given** 유효 입력 보드(예: PRD에 있는 스몰 케이스). **When** `solution(grid)` 호출. **Then** 반환 배열 **길이 6**, 요소 순서 `[r1,c1,n1,r2,c2,n2]`. | **출력 스키마 불변:** UI 계약과 동일하게 6-튜플 형태를 강제한다. |
| **LOG-RED-04b** | **Given** 동일 보드. **When** `solution` 호출. **Then** 모든 `r*, c*` ∈ [1,4] (1-index). | **좌표 체계 불변:** 로직 레이어에서도 UI와 동일한 1-index를 보장한다. |
| **LOG-RED-04c** | **Given** “첫 빈칸에 **작은** 후보를 먼저 넣었을 때” 마방진이 **성립하지 않는** 보드(또는 알려진 역전 케이스). **When** `solution` 호출. **Then** 알고리즘 계약에 따라 **reverse(큰 값 먼저 등)** 시도 후에만 성공하거나, 단일 정답이면 그에 맞는 한 쌍의 배치만 허용. **구현 전에는** 기대 동작을 고정한 assertion으로 **항상 실패**하게 설계. | **탐색 순서·백트랙 계약:** “small → first blank, 실패 시 reverse” 정책이 결과에 반영되는지 보호한다. |
| **LOG-RED-04d** | **Given** 솔버가 존재한다고 가정할 때, 반환 `(n1, n2)`가 `find_not_exist_nums`와 **집합적으로 일치**해야 함(순서는 `[n1,n2]`가 첫/둘째 빈칸에 대응). **When** `solution` vs `find_not_exist_nums` 교차 검증. **Then** 불일치 시 실패. (구현 없으면 `find_not_exist_nums` 자체가 없어 RED.) | **해 일관성:** 답으로 채운 두 수는 반드시 “누락된 두 수”와 같아야 한다. |

> **RED 상태:** `find_blank_coords`, `find_not_exist_nums`, `is_magic_square`, `solution` 본문이 없으면 import/NameError 또는 빈 구현으로 **전부 실패**해야 하며, 이것이 요구한 “오직 RED”이다.

## 1. 한 번만: 가상환경·설치

```bash
python -m venv .venv

pip install -e ".[dev]"
```

## 2. 테스트 실행

전체:

```bash
cd "c:/DEV/MagicSquare _1004"
pytest
```

자주 쓰는 옵션:

```bash
pytest -v                    # 테스트 이름·통과/실패 자세히
pytest --tb=short            # 실패 시 짧은 트레이스백
pytest tests/magicsquare/    # Magic Square RED 스켈레톤만
pytest -k "user"           # 이름에 user 포함된 테스트만
```

`pyproject.toml`에 `testpaths = ["tests"]`, `pythonpath = ["src"]`가 있어서 루트에서 `pytest`만 쳐도 `tests/`가 돌아가고 `src/magicsquare` import가 됩니다.

## 3. 커버리지

`dev` 옵션에는 **pytest만** 있고 **`pytest-cov`는 없음**이므로 커버리지 쓸 때만 추가합니다.

```bash
pip install pytest-cov
pytest --cov=magicsquare --cov-report=term-missing
```

- **`--cov=magicsquare`**: `src/magicsquare` 패키지 기준 라인 커버리지  
- **`term-missing`**: 터미널에 **안 탄 줄 번호**까지 출력  

HTML 리포트가 필요하면:

```bash
pytest --cov=magicsquare --cov-report=html
```

생성된 `htmlcov/index.html`을 브라우저로 열면 됩니다.

## 4. 참고

- PRD/README에는 Domain **≥ 95%** 등 목표가 있을 수 있어, 나중에 `--cov-fail-under=95` 같은 **하한 게이트**를 CI에 붙일 수 있습니다.  
- 지금 Magic Square RED 스켈레톤은 `pytest.fail("RED")`라 **전부 실패**하는 것이 정상입니다. 커버리지는 **구현 코드가 생긴 뒤** 의미가 큽니다.
